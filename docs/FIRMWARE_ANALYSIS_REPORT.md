# Slimbook GOS07 EC 固件静态逆向分析报告

## 1. 执行摘要

本报告对 `GXxHXxx_21.200`、配套 UEFI 刷写程序 `ifux64.efi` 和四个 64 KiB 代码块进行独立静态复核。旧有分析文档未作为结论来源；结论来自原始文件、反汇编和可复现的字节级检查。

主要结论如下：

1. `GXxHXxx_21.200` 是一个**无外层封装头的 256 KiB 原始 Flash 镜像**，可以精确切成四个连续 64 KiB block。block 0/1 是主 EC 的 banked 8051/8052 程序，block 2/3 从 `0x20000` 开始构成自带复位向量的 ITE8850 PD 固件。
2. 镜像并非“后半整体为 `FF`”。四个 block 都含大量有效数据；`FF` 主要是链接/地址空间空洞。最长连续 `FF` 区间是 `0x2D26F–0x2FFFF`，长度 11,665 字节。
3. changelog 中的 `CHKSUM 0x2124D9E` 精确等于镜像全部 262,144 字节的无进位整数和。这是偶然错误检测用的**非密码学加法校验和**，不能提供来源认证或防篡改能力。
4. `ifux64.efi` 是 ITE Flash Utility 2.0.3。它经传统 EC I/O 端口进入 follow/ISP 模式，读取 SPI ID，以 64 KiB 为单位擦除，确认擦除结果为 `0xFF`，写入数据，再从 Flash 读回逐字节比较。配套命令行明确允许任意文件名、任意烧录偏移和是否复位。
5. 在刷写器可执行文件中未发现固件签名解析、哈希、证书、公钥或镜像型号/版本白名单检查；PE Security Directory 也为空。输入路径只看到打开文件、取得文件长度和载入缓冲区，然后进入裸 Flash 流程。
6. 因而，**主机端更新工具没有数字签名验证的可信证据，且有很强证据表明其接受原始任意字节镜像**。但纯静态分析不能排除芯片 OTP/fuse、板级写保护或 EC 内部尚未触发的认证状态机。只有修改镜像的受控真机试刷或总线抓取才能把“高概率可刷”提升为“已实测可刷”。
7. 固件具备修改风扇、充电阈值、功能键和电源策略的技术基础；风险同时很高。主 EC 和 PD 共用一份整片镜像，错误的 bank、向量、校验/配置、PD 电源策略或断电均可能导致不开机、无法充电、USB-C 异常，乃至只能外接编程器恢复。
8. 后续真机只读实验取得 JEDEC ID `20 40 14 20`，并结合实物确认器件为 XMC XM25QH80（8 Mbit/1 MiB），完成 1 MiB 双读 dump。设备前 256 KiB 与官方镜像逐字节一致，`0x40000–0xFFFFF` 全部为 `FF`；这把镜像布局和真机 Flash 映射从静态推断提升为动态确认。详见 `docs/hardware_validation.md`。

### 结论置信度

| 判断 | 置信度 | 依据 |
|---|---:|---|
| 四个连续 64 KiB block | 高 | 文件长度、切片哈希、刷写器 `0x10000` 分块逻辑 |
| block 0/1 为主 EC、block 2/3 为 PD | 高 | 两套独立向量；`0x20040` 的 PD 自标识 |
| updater 不做主机端数字签名校验 | 高 | 完整字符串、PE 元数据和关键控制流均不见相关路径 |
| updater 做擦除/写入/读回比较 | 高 | 可执行文件控制流和状态字符串直接对应 |
| 可刷任意修改镜像 | 中高 | 裸文件输入和无签名路径；尚缺真机实验，不能排除 OTP/硬件锁 |
| EC 启动时完全没有 secure boot | 中低 | 当前镜像中未见明显验签实现，但 ROM/OTP 行为无法由镜像证明 |
| 官方 256 KiB 等于真机有效 Flash 内容 | 高 | 1 MiB 双读 dump；前 256 KiB 逐字节一致，后 768 KiB 全 `FF` |

## 2. 范围、方法与限制

### 2.1 分析对象

| 文件 | 大小 | SHA-256 |
|---|---:|---|
| `GXxHXxx_21.200` | 262,144 | `34c050d30772da07ef262fc7016e0677b9b1b4cdcd90cf43d93f0f15bf6a38c2` |
| `ifux64.efi` | 94,784 | `b013ddd898061a96bd10c8faffeb0b54300e5be4dcd933728b8a927b759f362d` |
| `F.nsh` | 29 | `299cd96a20a25f28013e625a4a24a70daae5a197a7457f9a5d79f8bd5f28eaae` |
| `CHANGELOG_EC.txt` | 2,428 | `411cd57387ea941d0c9b0cd93ffafb6824cd962aba4ea05b6ba1684305329e05` |

版本信息来自 changelog：EC `2.12.00.000`，日期 `2025-08-01`，PD FW `v27`。主镜像自身在 `0x50` 存在 `ITE EC-V14.6`，PD 镜像在 `0x20040` 存在 `ITE8850-PD` 和 `V0.27`。这几个版本字段属于不同层次，不应互相替代。

### 2.2 方法

- 字节级核验大小、哈希、切片一致性、加法校验和和 `FF` 连续区间；
- 检查各 block 的入口、向量表、字符串及 8051 指令流；
- 对 `ifux64.efi` 检查 PE 头、节、字符串和 x86-64 控制流；
- 将可执行文件中的状态文本与实际分支、循环、I/O 指令交叉验证；
- 对安全结论采用“发现的正面证据优先”，不以单纯“没搜到字符串”证明不存在。

### 2.3 限制

初始静态分析没有运行刷写程序或操作真机。后续已通过 Linux 只读工具取得 JEDEC ID 和双读一致的 1 MiB dump，但仍未执行擦除/编程、抓取 LPC/eSPI/SMBus/SPI 物理信号，也没有读取 IT5571/ITE8850 的 OTP、fuse 或外部 Flash 写保护脚。因此真机实验确认了读取通道和 Flash 内容布局，仍不能证明写入许可或芯片 ROM 内部不存在未公开保护。

## 3. 镜像格式与四块布局

### 3.1 镜像不是带 header 的更新容器

文件偏移 `0x00000` 直接以 8051 `LJMP 0x0070`（字节 `02 00 70`）开头，随后就是中断跳板。偏移 `0x20000` 同样直接以 `LJMP 0x83B7`（`02 83 B7`）开始另一套向量。文件头没有长度、签名目录或常见容器 magic；因此这是烧录地址从 0 开始的原始地址空间镜像，而非“header + payload”格式。

### 3.2 block 映射

| block | 文件范围 | SHA-256 | 非 `FF` 字节 | 最后一个非 `FF` 偏移 | 解释 |
|---:|---|---|---:|---:|---|
| 0 | `0x00000–0x0FFFF` | `41cc767b…e1c6` | 58,337 | `0x0FF00` | 主 EC 公共区/低 bank，含复位及中断向量 |
| 1 | `0x10000–0x1FFFF` | `a19a0b53…5626` | 54,721 | `0x1E97A` | 主 EC 扩展 bank 代码与表 |
| 2 | `0x20000–0x2FFFF` | `e9cda656…1570` | 52,580 | `0x2D26E` | PD 低 bank，含独立向量和身份字符串 |
| 3 | `0x30000–0x3FFFF` | `be291f2b…375a` | 52,971 | `0x3EC37` | PD 扩展 bank 代码与表 |

已有的 `samples/disasm/main-bank0.bin`、`main-bank1.bin`、`pd-bank0.bin`、`pd-bank1.bin` 与原始镜像对应 64 KiB 范围逐字节完全相等。这个一致性仅证明切片正确，不自动证明旧反汇编中的语义标注正确。

### 3.3 `FF` 填充的实际分布

主要连续空洞如下：

| 范围 | 长度 | 所属 block |
|---|---:|---:|
| `0x2D26F–0x2FFFF` | 11,665 | 2 |
| `0x36842–0x37FFF` | 6,078 | 3 |
| `0x1E97B–0x1FFFF` | 5,765 | 1 |
| `0x0E928–0x0FDFF` | 5,336 | 0 |
| `0x3EC38–0x3FFFF` | 5,064 | 3 |
| `0x16E92–0x17FFF` | 4,462 | 1 |

这说明固定 256 KiB 地址空间内存在链接空洞和 bank 边界填充，但不能把后 128 KiB 解释为未使用空间：PD 两个 block 均有超过 52,000 个非 `FF` 字节。

### 3.4 changelog 校验和

对全部字节计算：

```text
sum(image[i], i=0..262143) = 34,753,950 = 0x2124D9E
```

结果与 changelog 完全一致。该值大于 32 位截断需求且没有密钥参与，算法就是简单 byte sum。攻击者修改一个字节后，可通过调整另一个非关键字节恢复相同总和；它只能检测常见传输/文件损坏，不能认证固件。

## 4. 主 EC 固件

### 4.1 处理器和编译模型

指令集是 MCS-51/增强 8052 风格，广泛使用 `MOVX @DPTR` 访问扩展寄存器空间。结合机器硬件信息，主 EC 被识别为 ITE IT5571；二进制本身只直接自标识为 `ITE EC-V14.6`，因此具体料号仍以板级器件标识为最终依据。

代码超过 64 KiB，使用固定低地址入口和 bank 切换跳板。block 1 中的调用目标继续使用 16 位 8051 地址语义，说明四个 block 不是四个顺序执行的普通“阶段”，而是烧录器按 Flash 容量划分的块；CPU 侧通过 bank/address-extension 机制访问。

### 4.2 复位与初始化

主 EC 的复位向量位于 `0x00000`：

```text
0000: LJMP 0070h
0070: MOV SP,#C0h
0073: MOV DPTR,#1001h
0076: MOV A,#3Fh
0078: MOVX @DPTR,A
```

随后调用多项硬件初始化例程，并进入典型 C51 运行时初始化：从代码区表读取长度和目标地址，清零或复制内部/XRAM 数据，最后转入应用入口。该形态说明镜像是可直接启动的完整程序，而不是数据包。

### 4.3 中断与事件调度

低地址向量通过 `LJMP` 转到实际处理函数。可确认的入口包括：

| 向量 | 目标 | 静态行为 |
|---|---:|---|
| RESET `0x0000` | `0x0070` | 系统和 C 运行时初始化 |
| INT0 `0x0003` | `0x0532` | 立即返回，当前映像未使用 |
| Timer0 `0x000B` | `0x0533` | 重装计时器并置位任务标志 |
| INT1 `0x0013` | `0x0559` | 读取主机/EC 邮箱并分派命令 |
| Timer1 `0x001B` | `0x05BF` | 超时/周期事件处理 |
| Serial `0x0023` | `0x05E7` | 当前入口立即返回 |
| Timer2 `0x002B` | `0x05E8` | 当前入口立即返回 |

主循环在无待处理标志时写 `PCON=0x01` 进入 8051 IDLE，由中断唤醒。Timer0 初始化使用 `TMOD=0x11`，其 ISR 推动按位任务调度器；调度器清除标志后调用相应 worker。这是常见的短 ISR + cooperative worker 架构。

### 4.4 主机邮箱

INT1 路径读取扩展地址 `0x1110` 的命令值，至少直接分支处理 `0x22`、`0x23`、`0x24`，其他值通过表驱动路径。代码还会更新 `0x1100–0x1151` 一带的状态/握手寄存器并通知主机。

这证明主 EC 与主机固件/操作系统之间存在命令邮箱，但当前静态证据不足以为三个命令赋予准确的人类语义。报告不沿用旧文档中未经动态验证的命令命名。

### 4.5 可识别功能域

从调用密度、寄存器访问和数据表可以确认固件覆盖以下笔记本 EC 职能：

- 双通道 SMBus/I²C 风格控制器访问；
- 电池、适配器和充电状态管理；
- 两路风扇/PWM 及转速相关表；
- 温度/ADC/GPIO 状态采集；
- 键盘/Fn 键与主机事件；
- 睡眠、唤醒和低功耗状态切换；
- 看门狗、计时器和周期任务。

例如代码直接访问 `0x1652/0x1654` 一带的两路 PWM 状态、`0x170x` 的电源/充电状态以及 `0x180x/0x1F0x` 两组串行总线寄存器。这里的“功能域”判断可靠，但在没有 IT5571 官方寄存器手册和板级原理图时，不应把每个位的具体含义视为已证实。

## 5. PD 固件

### 5.1 独立程序边界

偏移 `0x20000` 是第二套完整 8051 向量表：

```text
20000: LJMP 83B7h    ; RESET
20003: LJMP 0056h    ; INT0
2000B: LJMP 0094h    ; Timer0
20013: LJMP 00B2h    ; INT1
2001B: LJMP 00F0h    ; Timer1
20023: LJMP 010Eh    ; Serial
```

紧邻向量的 `ITE8850-PD`、`V0.27` 是强身份依据。它与主 EC 有各自的复位入口、栈初始化、中断保存/恢复和应用主循环，因此是独立 PD 控制器固件，而非主 EC 的普通资源段。

### 5.2 启动和代码组织

复位目标 `0x83B7` 执行内部 RAM/XRAM 初始化、设置栈并调用运行时/硬件初始化，然后跳入应用入口。PD block 同样采用 banked 地址组织。block 2 和 block 3 中存在部分位于相同 bank 内偏移的相似调试字符串/公共例程，这是 banked 固件复制公共常量或代码的表现，不能据此把它们误判为重复无效数据。

### 5.3 可见协议功能

PD 镜像直接包含：

- `Protover:01.00`、`UsbPdVer:01.00`、`Hotfix:0000`；
- source-role 尝试成功/失败调试信息；
- I²C timeout、arbitration lost 和状态计数诊断；
- 多组状态机、计时和消息缓冲代码。

这些证据支持 ITE8850 负责 USB-C/USB-PD 协议及与板上器件通信。仅凭字符串不能确认其支持的全部 PD revision、PDO、电压档位或保护阈值；修改这些内容需要进一步恢复状态机和配置表结构。

## 6. UEFI 刷写器逆向

### 6.1 基本属性

`ifux64.efi` 是 x86-64 PE32+ EFI application：

- 入口 RVA `0x2E0`；
- EFI subsystem 10；
- 无普通 PE import table，符合 EDK2 应用通过 EFI System Table/Protocol 指针调用服务的方式；
- PE CheckSum 为 0；
- Security Directory RVA/Size 均为 0，即该 EFI 文件本身也没有嵌入 Authenticode 证书表；
- 内部版本字符串为 `ITE Flash Utility Version : 2.0.3`。

“EFI 文件未签名”与“平台 Secure Boot 是否允许启动它”是两个不同问题；外部启动介质、固件策略或 shim 仍可能限制其执行，但这不构成 EC 镜像验签。

### 6.2 命令行和输入模型

工具自身给出的接口是：

```text
ifu <ec filename> [burn offset] [reset]
ifu ec.bin
ifu ec.bin 0x400000 0
ifu ec.bin 0 1
```

配套 `F.nsh` 实际执行：

```text
IFUX64.efi GXxHXxx_21.200 0 1
```

即从 Flash 偏移 0 烧录并在结束后复位。控制流打印 `filename`、`gOffset`、`gReset`，打开指定文件，取得 `File Size`，分配/填充缓冲区。没有看到必须匹配特定文件名、magic、EC 版本或板型的分支。

### 6.3 EC/Flash 通道

代码使用 x86 `IN/OUT` 指令访问传统键盘控制器/EC 端口 `0x66`（状态/命令）和 `0x62`（数据）。可见流程包括：

1. 向键盘/EC 发送 `0xAD`、`0xDC`；
2. 等待 input/output buffer 状态位；
3. 读取 EC ACK，并与 `0x33` 比较；
4. 进入 follow mode；
5. 读取四字节 SPI Device ID；
6. 根据 ID 匹配 SST、Winbond、Atmel、ST、Spansion、MXIC、AMIC、EON、ESMT、ITE、GigaDevice、PMC 等厂商及相应操作例程。

这是一条通过主 EC 桥接 SPI Flash 的低级 ISP/follow-mode 路径，不是向运行中 EC 发送高层“已签名 capsule”。

### 6.4 擦除、写入和验证

关键流程可由控制流直接确认：

1. 根据文件长度计算 block 数，block 基数为 `0x10000`；256 KiB 文件因此得到 4 个 block。
2. 对当前 block 发送对应 SPI 擦除命令，并等待完成。
3. `Erase Verify` 从该 block 起始位置读回，期望每个字节均为 `0xFF`；不一致时打印偏移计数并失败。
4. `Programing` 以页/小块循环发送地址和原始文件字节。
5. `Verify` 再次从 Flash 读取，逐字节与文件缓冲区比较；不一致时打印 `[address]=value` 和 `Verify Fail`。
6. 移动到下一个 64 KiB block，直至文件长度覆盖完成；根据 `reset` 参数退出模式并复位。

因此用户观察到的“四次 erase/write/verify”不是四次重试，而是 256 KiB 文件除以 64 KiB 得到的**四个烧录 block**。程序内部也存在等待、厂商差异处理和失败分支，但当前证据不支持把数字 4解释为统一 retry count。

## 7. 安全机制评估

### 7.1 已确认存在的完整性措施

- 发布记录中的全镜像 byte sum；
- 擦除后逐字节确认 `0xFF`；
- 写入后把 Flash 内容与输入缓冲区逐字节比较；
- SPI Device ID/厂商识别；
- EC 进入 ISP/follow mode 的命令及固定 ACK。

这些措施解决“烧录到了正确芯片、擦干净、写入没有位错误”等可靠性问题，不验证发布者身份。

### 7.2 未发现的主机端安全措施

在 `ifux64.efi` 和镜像中未发现以下可验证结构或代码路径：

- SHA-2/SHA-3 等固件摘要计算；
- RSA、ECDSA/EdDSA 签名校验；
- X.509/DER/PE certificate 解析；
- 固件签名字段、签名长度或公钥表；
- 固件型号、板型、版本回滚策略或版本白名单；
- challenge-response、nonce 或基于密钥的 ISP 解锁协议。

可执行文件中的通用 C/EDK2 errno 文本含 `Authentication error`、`Need authenticator` 等词，但它们位于标准运行库错误字符串表中，没有证据连接到刷写控制流，不能视为固件认证功能。

### 7.3 为什么“没找到公钥”还不是绝对证明

公钥不必以 PEM 或可读字符串存在；它可以是裸大整数/椭圆曲线点，也可能固化在芯片 mask ROM 或 OTP。更重要的是，写入许可可能由硬件写保护、strap、板级 GPIO 或 EC 当前状态决定。因此本报告的严格表述是：

> 更新包和 ITE Flash Utility 2.0.3 未实施可见的主机端密码学固件认证；其数据路径表现为任意文件字节的裸 SPI 烧录。是否存在芯片 ROM/OTP 或板级写保护，尚未通过动态实验排除。

### 7.4 可修改/可刷性判断

综合证据，工具接受修改后 256 KiB 原始镜像的概率很高。简单 byte sum 只出现在 changelog，刷写器可见路径没有比较 `0x2124D9E`；即使下游另有比较，该和也可容易重算或补偿。

建议将状态分为：

| 层级 | 当前状态 |
|---|---|
| 修改文件并被工具读取 | 已由接口和控制流支持 |
| 工具把任意字节送入 Flash 编程路径 | 静态证据强 |
| EC/Flash 在目标机器上接受写入 | 原版更新应可完成，但修改版尚未实测 |
| 修改后主 EC/PD 正常启动 | 未验证，取决于代码正确性及潜在 ROM 检查 |
| 可可靠恢复失败刷写 | 未验证，必须在实验前建立 |

## 8. EC mod 可行性与风险

### 8.1 较可行的修改方向

- 修改已定位的数据表，而不是直接重写复杂控制流；
- 调整风扇曲线/PWM 上下限；
- 调整充电上限、温度门限或性能策略；
- 修改 Fn 键映射和事件表；
- 在保留地址、长度和 bank 布局的前提下做小范围补丁。

changelog 已表明厂商历史版本确实修改过充电限制、低温开机、性能和 Fn 键，说明这些策略位于可更新固件中。但仅凭不同版本的文字记录无法定位具体字节；若能取得 2.05–2.10 的旧镜像，差分分析会显著提高定位可靠性。

### 8.2 主要风险

| 风险 | 后果 |
|---|---|
| 主 EC 复位向量、bank thunk 或初始化损坏 | 完全不开机，内置刷写路径也可能消失 |
| 风扇/温控表理解错误 | 过热、噪音、降频或硬件损伤 |
| 充电/适配器参数错误 | 无法充电、电池过流/过压保护触发 |
| PD 状态机或 PDO 配置错误 | USB-C 失效、错误供受电协商、外设风险 |
| 整片擦写中断电 | 四个 block 状态不一致，系统不可启动 |
| 仅备份文件、不备份设备原始 Flash | 丢失设备专属校准/配置（若存在） |
| 假定软件 verify 等于启动成功 | 字节写对了，但代码仍可能逻辑失效 |

### 8.3 实验前最低恢复条件

在进行任何修改版刷写前，应至少满足：

1. 从目标机器读取并保存整片 Flash 的原始 dump，且做两次读取哈希比对；
2. 确认 Flash 芯片型号、供电电压、封装、写保护脚和可接触测试点；
3. 准备兼容电压的外部 SPI 编程器、夹具或焊接恢复方案；
4. 分清主 EC 与 PD 的复位/供电关系，确保外部编程时主板不会反向供电；
5. 第一阶段只做可观察、可逆且不影响安全阈值的单字节/表项修改；
6. 保留原始镜像、修改镜像、差异清单、补丁脚本和每次烧录日志。

## 11. 最终判断

该更新包属于传统 ITE EC 裸 Flash 更新设计，而非带签名 capsule 的现代安全更新链。四次进度来自四个 64 KiB block；每块执行擦除、擦除校验、编程和读回比较。工具负责写入可靠性，但未见发布者认证。

就现有证据而言，制作并烧录修改版 EC 镜像在技术上很可能可行；最重要的未知项不再是“文件里有没有公钥”，而是目标机器是否有 ROM/OTP/板级写保护，以及失败后能否绕过失效 EC 直接恢复 SPI Flash。在建立硬件恢复手段之前，不建议把静态分析结论直接转化为真机写入实验。
