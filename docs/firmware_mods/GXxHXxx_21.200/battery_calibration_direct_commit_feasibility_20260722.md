## 结论

技术上可行，而且可以做成只改 3 字节的局部补丁：状态 5 已经先计算出本轮候选 factor，
随后才用旧 FCC 派生的上下界将它夹到每轮 `+/-3` factor（即约 `+/-300 mAh`）。在候选
计算返回后直接跳到现有提交块，即可保留 factor 写入、乘 100、发布 pending 和状态迁移，
同时跳过限幅比较。现有旧审计把提交函数写成 `C970`、汇合点写成 `C98F`；本次按完整镜像
逐字节复核后确认它们分别应为 `C96E` 和 `C98E`，以下地址以原始 opcode 为准。

但当前只建议把它作为**实验候选补丁**，不建议立即取代已实机验证的
`GXxHXxx_21.200-int1-r7-20260722.bin`。原因不是补丁难做，而是候选计算 helper 的异常输入
范围、精确舍入和 `0x0342/0x03C0` 镜像同步尚未完成动态验证。原厂的步进限制很可能同时
承担抗瞬态积分误差和防止单轮错误 FCC 跃迁的保护作用。

## 当前数据流

原始固件的状态 5 路径为：

```text
XRAM[0x0388:0x0389]（本轮 charged-capacity 累计）
  -> bank1:E965 capacity_ratio_helper（除以 100，factor 结果在 R7）
  -> 与 XRAM[0x039B] 上界、XRAM[0x039C] 下界比较
  -> accepted factor 写 XRAM[0x0342]
  -> factor * 100 写 XRAM[0x0340:0x0341]
  -> 设置 XRAM[0x03A1].bit1，状态进入 6
  -> bank1:DA8E 在镜像条件满足后发布 EC[0x0404:0x0405] FCC
```

边界在本轮开始时由 `bank1:C7BA` 从旧 FCC 生成：

```text
high = old_FCC / 100 + 3
low  = old_FCC / 100 - 3
```

因此要“直接改成校准获得的值”，应绕过**提交时的 clamp**，而不是修改常数 `3`。把常数
改大仍然保留人为范围，并会同时影响上下界溢出行为；删除边界初始化也没有用，因为提交
函数仍会读取旧/未定义的 `0x039B/0x039C`。

## 原始 opcode 与地址核对

以下字节来自官方镜像 block1，即 `samples/disasm/main-bank1.bin`。block1 在完整 256 KiB
镜像中的基址为 `0x10000`；该路径位于 bank1 的 banked window，不属于 fixed common。

```text
bank1 文件  CPU 地址      原始字节             解码
0x496E      bank1:C96E    90 03 88             MOV  DPTR,#0388
0x4971      bank1:C971    12 E9 65             LCALL E965
0x4974      bank1:C974    90 03 9B ...         上界/下界比较与选择
0x498E      bank1:C98E    90 03 42 EF F0       MOV DPTR,#0342; MOV A,R7; MOVX @DPTR,A
```

`LCALL` 目标由调用点原始 opcode `12 E9 65` 直接解码为 `0xE965`。调用发生在 bank1
window，目标也高于 `0x8000`，所以是 `bank1:E965`，不是 common 地址或其他 bank 的同名
偏移。

候选最小补丁为：

```text
完整镜像物理偏移 0x14974
原字节：90 03 9B                         MOV DPTR,#039B
新字节：02 C9 8E                         LJMP bank1:C98E
```

`LJMP` 目标同样由新 opcode `02 C9 8E` 直接解码。它位于 banked 区，执行期间 bank 选择
仍为 bank1，因此落到当前函数已有的提交基本块。`0x4977--0x498D` 会变成不可达旧 clamp
代码，无需覆盖，也不需要占用 common 洞。以已修复 INT1 R7 的镜像为基线时，block1 原本
未被修改，所以相同物理偏移和原字节仍成立；实际生成器仍必须校验输入 SHA-256 和原字节。

## 可行性与主要风险

静态控制流层面可行性高：候选已经在 `R7`，跳转目标就是原函数所有分支汇合的提交块，
没有新增栈使用、跨 bank thunk 或寄存器约定变化。后续发布流程也完全复用原代码。

行为层面的信心目前为中等，主要风险如下：

1. `R7` 是 8 位 factor，公开 FCC 的粒度仍是 100 mAh；“直接提交”是提交
   `floor/round(candidate / 100)` 的结果，不是保存未经量化的精确 mAh 值。
2. 原厂 clamp 能把一次坏会话的影响限制在 300 mAh。取消后，积分基线错误、遥测中断、
   会话方向误判或异常满电终点可在单轮造成大幅 FCC 跃迁。
3. helper 对极小值、超过设计容量的值以及除法舍入尚未完整验证。虽然正常 5200 mAh
   电池远未触及 u8 factor 上限，但错误累计仍可能产生不合理候选。
4. `DA8E` 要求 `0x0342 == 0x03C0` 才发布。现有流程能最终同步，但尚未证明大跨度变化是否
   会被电池遥测镜像拒绝、回滚或延迟；绕过 clamp 不改变这个条件，却可能暴露未测分支。
5. 当前实机记录证明过 `3700 -> 4000 mAh` 的上限命中，没有记录一轮候选远离旧 FCC 时
   直接提交后的完整状态 5、状态 6 和重启持久化过程。

## 建议的实验版本

不要直接删除所有保护。首个实验版本建议“跳过动态 `+/-300` clamp，但增加绝对合理性
门”，例如只接受 `10..design_capacity/100` 范围内的 factor；不满足则沿用旧 factor 或
终止本轮。这个方案需要额外代码洞和明确的 design-capacity 换算，改动面比 3 字节跳转大，
但能避免单次异常把 FCC 写成 0、数百 mAh 或超过设计容量。

如果目标是先验证原始候选是否可信，风险更低的顺序是：

1. 保持当前固件不变，用 `tools/watch_battery_calibration.py` 记录状态 5 前后的
   `0x0388:0x0389`、`0x039B/0x039C`、`0x0342/0x03C0` 和公开 FCC。
2. 从 `0x0388` 按固件 helper 的真实运算复算未限幅 factor；至少覆盖一次候选落在范围内、
   一次命中上界和一次命中下界。
3. 生成单独命名的实验镜像，例如
   `GXxHXxx_21.200-int1-r7-cal-direct-exp-20260722.bin`，不得覆盖当前已验证版本。
4. 首刷前保留两份一致原机 dump 和硬件恢复路径；刷后先检查空闲、充电中、满电发布及冷
   启动，不以一次 Linux `charge_full` 变化作为成功结论。

## 验收标准

- 固件 diff 除 INT1 R7 修复的 29 字节外，只新增物理 `0x14974--0x14976` 三字节变化；
- 从 opcode 确认 `0x14974: 02 C9 8E` 在 bank1 window 跳到 `bank1:C98E`；
- 状态 5 提交值等于同轮 helper 的未限幅候选，而不是 `0x039B/0x039C`；
- `0x0342`、`0x03C0`、`0x0340:0x0341` 和 `0x0404:0x0405` 最终一致；
- 状态 6 pending 能被消费，冷启动后 FCC 不回滚且不超过 DesignCapacity；
- AC 插拔、睡眠/唤醒、关机充电和遥测失败恢复不产生 0 或明显异常 FCC。

综上，3 字节直跳方案足以验证想法，静态上没有控制流障碍；但把它作为长期固件前，至少应
先证明未限幅候选在异常会话中仍有可靠绝对边界。更适合长期使用的版本是“直接接受正常
候选 + 绝对容量范围保护”，而不是完全无保护提交。
