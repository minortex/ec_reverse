# EC 固件二进制修复设计（2026-07-22）

本文记录最初的离线补丁设计；原始 256 KiB 镜像是唯一权威输入，`.d52` 中的
overlapping 标注必须回到原始字节核对。该设计后来生成并刷入
`firmware_mods/GXxHXxx_21.200/int1-r7-20260722/GXxHXxx_21.200-int1-r7-20260722.bin`，
初步实机结果为 17600 mV 保持稳定且
暂未观察到异常。当前镜像身份、精确差异和后续维护要求见
`GXxHXxx_21.200_int1_r7_firmware_notes_20260722.md`。

## 结论

目前最小、可验证的根修复是：在 INT1 ISR 中保存并恢复 bank2 的 R7。不能把
`MOV PSW,#0x10` 改成 `#0x18`：原始 bank1 文件偏移 `0x1A03`（物理 Flash
`0x11A03`、CPU 逻辑地址 `bank1:0x9A03`）存在可达的 `MOV 0x1C,@R0`，即
bank3 的 R4 槽被正常代码使用。

保存全部 R0--R7 的方案理论上更完整，但当前不能证明 INT1 深层调用的最大栈深，
因此只能作为第二阶段实验补丁。复位时 `SP=0xC0`，硬件入口已占 2 字节，原 ISR
又保存 ACC/B/DPH/DPL/PSW 五字节；额外 8 字节可能触及最深调用路径的栈上限。

## 推荐的 R7-only 离线补丁

原始物理偏移与字节如下（主 EC block0，逻辑地址相同）：

```text
0559: c0 e0 c0 f0 c0 83 c0 82 c0 d0 75 d0 10
0566: 30 6e 05                         ; JNB 2D.6,056E
05B2: d2 6c d0 d0                       ; SETB 2D.4; POP PSW
7E6A: ff ...                            ; 代码洞
```

把 `0566` 的 3 字节改为 `LJMP 7E6A`，把 `05B2` 的 3 字节改为
`LJMP 7E78`，并在 `7E6A` 写入：

```text
7E6A: c0 17             ; PUSH direct 17h (bank2 R7)
7E6C: 30 6E 06         ; JNB 2D.6,7E75
7E6F: 12 14 C8         ; 原特殊分支 LCALL 14C8
7E72: 02 05 B2         ; 进入统一尾部（已重定向）
7E75: 02 05 6E         ; 普通邮箱分支从原入口继续
7E78: D0 17             ; POP direct 17h
7E7A: D2 6C             ; 原 SETB 2D.4
7E7C: D0 D0             ; 原 POP PSW
7E7E: 02 05 B6         ; 继续原 ISR 尾部 POP DPL...
```

这样两个 INT1 分支都在返回主线前恢复被中断上下文的 R7；普通分支仍从
`056E` 执行原邮箱分派，特殊分支仍调用 `14C8`。补丁只占用 `7E6A--7E80`
的一小段连续 `FF` 区域，不改变向量和其它 bank。

`14C8` 必须按原始 opcode 解码：`90 A2 74 02 11 00` 是
`MOV DPTR,#A274; LJMP 1100`。`1100` 选择 bank0，因此它进入的是
`bank0:A274`（原始字节为 `RET`），不是 bank1 同地址代码。此前把该路径接到
bank1 `A274/A3D9` 的结论无效，但不影响本补丁，因为补丁逐字保留了原
`LCALL 14C8`。

注意 `PUSH/POP 17h` 必须发生在 `MOV PSW,#10h` 之后、`POP PSW` 之前；此时
`17h` 才是 bank2 的 R7。不能用 `PUSH 07h`，也不能在切换 PSW 前保存 17h。

## 完整 R0--R7 方案（暂不刷）

若后续动态证据表明 ISR 还破坏 R0--R6，可将 `7E6A` 改为依次
`PUSH 10h` ... `PUSH 17h`，尾部逆序 `POP 17h` ... `POP 10h`，再执行
`SETB/POP PSW/原尾部`。这需要 0x33 字节洞内代码和额外 8 字节栈深，必须先取得：

1. INT1 从硬件入口到最深 `LCALL` 的实际最大 SP；
2. 所有跨 bank wrapper 是否会在返回前依赖 ISR 的 R0--R7 副作用；
3. 复位、邮箱命令 `22/23/24`、`2D.6=1` 特殊路径的回归结果。

在无法取得这些证据时，R7-only 是风险更低且直接针对已观察 `0x1B` 的版本。

## 为什么不采用其它补丁

- 改 `PSW` 到 bank3：已被 `0x1C` 的真实代码引用否决。
- 只屏蔽 D1D1 的 INT1：可减小窗口但要处理 pending EX1，且不能修复其它共享
  bank 的 ISR 重入点。
- 只改 `0x0A54` 或主机重复写 `0x0522`：下一个周期仍会重新产生错误分档。

## 离线验证步骤

```sh
sha256sum samples/GXxHXxx_21.200
cp samples/GXxHXxx_21.200 /tmp/GXxHXxx_21.200-int1-r7-20260722
# 仅按上表修改 0566、05B2、7E6A--7E80
python3 tools/firmware_tool.py diff \
  samples/GXxHXxx_21.200 /tmp/GXxHXxx_21.200-int1-r7-20260722
```

期望差异只有 29 个字节（两个 3 字节跳板和 23 字节洞内代码），且全部位于
block0 的上述三个区间。随后应重新反汇编
block0，人工确认每个 `LJMP` 目标和相对路径；再对四个 64 KiB block 做逐字节
verify。当前刷写器证据显示它做擦除校验和写后逐字节比较，但没有发现发布签名或
强制 byte-sum 校验；原镜像的记录值为 `sum=0x2124D9E`、`sum16=0x4D9E`。

## 烧录前的必要门槛

必须保留双份原机 256 KiB dump、确认两次 dump SHA-256 相同，并准备已验证的
硬件恢复路径。没有这些条件时，不应把任何实验补丁写入 EC；主机侧持续写
`0x0522/0x0523` 只能作为临时规避，不能替代固件修复。
