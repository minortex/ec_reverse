# 充电电压目标卡在 16.6 V 的固件分析

## 结论

当前机器同时存在三个不同的电压量，必须先分开：

| 对象 | 地址 | 当前实测 | 含义 |
|---|---:|---:|---|
| 实时电池包电压 | `EC[0x0438:0x0439]` | `16396 mV` | Fuel Gauge 遥测；Linux `voltage_now` 也为 `16396000` uV |
| 电池侧未降额请求 | `XRAM[0x030E:0x030F]` | `17600 mV` | 电池遥测块提供的 charger voltage 基准 |
| EC 保存的充电目标 | `EC[0x0522:0x0523]` | `16600 mV` | EC 交给 charger 队列的期望值，不是 charger readback |

所以“卡在 16.6 V”有两层含义：

1. `EC[0x0522:0x0523]` 确实长期保存 `16600 mV`，没有跟随当前 `17600 mV` 基准重算；
2. 用户看到的 `16.396 V` 是当时正在充电的实时包电压，不是一个 16.396 V 限制值。

对第一个问题，当前已经定位到
`CODE:0xD1D1 update_battery_aging_charge_voltage_derating` 允许形成 **stale target** 的控制流机制：

- 本机充电采样中，`XRAM[0x050B].7 == 0`，函数清除应力后在 `CODE:0xD1F6` 直接 `RET`；
- 本机放电采样中，`XRAM[0x050B].7 == 1`，但 `EC[0x0490].0 == 0`，函数又在最终计算前返回；
- 清除的是 `XRAM[0x09C7:0x09CA]` 应力状态，不是 `EC[0x0522:0x0523]` 保存目标；
- `D1D1` 内写回 `0x0522:0x0523` 的路径位于函数尾部 `CODE:0xD3EC–0xD422`，本次观察到的两个稳态组合都到不了。

因此，重新刷写 EC、应力清零和电压目标恢复是三件不同的事。按当前 gate 和约 720 ms 的
周期调用路径，应力状态随后应被清零，而 16600 mV 目标仍可保留。由于没有同步直接读取
`0x09C7:0x09CA`，本文不把“采样时应力已经为零”列为独立实测事实。

## 调查范围与证据

分析基于：

- 当前 2.12 `main-bank1.bin` 的原始机器码；
- `samples/disasm/main-bank1.d52` 的 8051 反汇编；
- `build/readable-main1/main-bank1.bin.c` 的 Ghidra 伪 C；
- `/home/texsd/codes/mech-forza-control/tools/ec_rw.py` 的只读实机采样；
- Linux `/sys/class/power_supply/BAT0` 的同步状态。

`.d52` 使用 `org 0-1h`，其中 `dptr_NNNN` 符号比机器码立即数小 1。本文地址均已按原始
`MOV DPTR,#imm16` 修正；关键结论不直接依赖未修正的符号后缀。

## 实机时间线

### 满电接 AC

初次只读采样：

```text
base target       0x030E:0x030F = 17600 mV
saved target      0x0522:0x0523 = 16600 mV
pack voltage      0x0438:0x0439 = 16419 mV
RSOC              0x04AB        = 100%
direction high    0x050B        = 0x00
power state       0x0490        = 0x0F
battery session   0x0497        = 0x45 (bits 6, 2, 0 set)
battery mode      0x07A6        = 0x00 (Normal)
cycle count       0x04A6:0x04A7 = 57
```

`0x050B.7 == 0`，所以周期函数会清除高压应力并提前返回。此时保存目标仍为 16600 mV。

### 断开 AC 放电

放电到约 98% 时：

```text
BAT0 status                         Discharging
AC0 online                          0
discharge current 0x0434:0x0435    817 mA
pack voltage      0x0438:0x0439    16053 mV
direction high    0x050B           0xFA (bit 7 = 1)
power state       0x0490           0x0E (bit 0 = 0, bit 2 = 1)
saved target      0x0522:0x0523    16600 mV
```

放电时方向 gate 开放，但电源 bit 0 关闭，最终目标仍不重算。包电压还低于 4S 高压门槛
`4 * 4100 = 16400 mV`，所以这段放电也不会增加高压应力。

### 接电过渡与正常充电

接电过程中连续读取到：

```text
放电稳态    0x050B = 0xFC, 0x0490 = 0x0E
切换时      0x050B = 0x00, 0x0490 = 0x0F
充电稳态    0x050B = 0x02/0x03, 0x0490 = 0x0F
```

随后同步采样：

```text
BAT0 status                         Charging
BAT0 current_now                    374 mA
BAT0 capacity                       97%
BAT0 voltage_now                    16396000 uV
EC pack voltage                     16396 mV
EC base target                      17600 mV
EC saved target                     16600 mV
0x050B                              0x01 (bit 7 = 0)
0x0490                              0x0F (bit 0 = 1, bit 2 = 1)
```

这同时表明：

- `0x050A:0x050B` 是以 mA 为单位的 16 位有符号电池电流：正值充电、负值放电；
- 本次普通充电采样中该符号/方向 gate 关闭；
- 接电过渡没有让 `0x0522:0x0523` 从 16600 更新到 17600；
- 16.396 V 是实时包电压，因为 EC 和 Linux 在同一时刻给出完全一致的值。

### `0x050A:0x050B` 已可确定为 signed battery current

后续离电、插电、离电、再插电的高字节时间线重复出现：

```text
离电稳态：0x050B = 0xFD/0xFC/0xF9（有符号负数）
插电过渡：0x050B = 0x00/0x01
充电稳态：0x050B = 0x06/0x07（有符号正数）
```

这不再只是实测相关性。`CODE:0x2CF3–0x2D3A` 读取完整 word；若 `0x050B.7` 为 1，便对
它取 16 位补码绝对值，最后把幅值写入公开的 `EC[0x0434:0x0435] battery_discharge_current`。
因此可以确定：

```c
signed_battery_current_ma = (int16_t)(XRAM[0x050B] << 8 | XRAM[0x050A]);
EC[0x0434:0x0435] = abs(signed_battery_current_ma);
```

同步样本也验证了单位：充电 `374 mA = 0x0176`，高字节为 `0x01`；放电 `817 mA` 在内部
表示为 `-817 = 0xFCCF`，高字节为 `0xFC`。只读取高字节只能得到区间：例如 `0x07`
表示正电流 `1792–2047 mA`，`0xFD` 表示负电流 `-768–-513 mA`，不能由高字节恢复精确值。

`CODE:0x5936` 一类的内部发布入口会把同一个输入 word 复制到包括 `0x050A:0x050B` 在内
的四组状态槽：`0x050A:0x050B`、`0x050C:0x050D`、`0x050E:0x050F` 和
`0x0510:0x0511`。`0x2CF3` 发布公开幅值前还会比较 `0x050B` 与 `0x050D` 两个高字节，
相等才读取主副本，说明这些副本至少部分用于避免异步更新时取得撕裂的 16 位值，而不是
四种不同物理电流。更上游的 Smart Battery/遥测事务尚未完全恢复。结合 SBS Current 的 signed
mA 语义，它极可能直接或经轻量缓存来自电池电流遥测，但“对应哪一次总线命令和刷新状态”
仍需继续追踪。

这个命名还澄清了 `D1D1` 的 gate 极性：老化应力只在电池电流为负，即电池正在对系统
放电时进入后续路径；零电流或正向充电会清空应力并返回。这一极性与通常的“充电高压热
应力累计”直觉相反，但它是当前机器码和多轮实测共同给出的执行事实，不能再把 bit 7
泛称为未知方向位。这里的反常可能来自通用模板与项目电流极性不匹配、不可达保留模板，
或厂商条件分支错误；仅凭本镜像不能判断源代码意图，也不能把它包装成合理的老化模型。

### `0x0497.0` 是电池数据会话资格位

对真实地址 `0x0497` 的全部直接引用复核后，不能再把整个 byte 称为 aging gate。
`0x0497` 是由多个电池状态机共享的 flags byte；`D1D1` 只消费其中 bit 0。

bit 0 的生产路径具有明确的生命周期：

```text
CODE:0x4268  完成电池缓存同步及初始化调用后：0x0497 = 0x81
              bit 0 = 数据会话资格成立
              bit 7 = 同时进入一个瞬态/busy 阶段

CODE:0x427B  重启相邻电池状态机时：0x0497 = 0x00

后续路径     可单独清 bit 7，bit 0 继续保留
```

bit 0 还被多个独立消费者用于选择电池数据表、允许电流/容量相关处理或切换兼容路径，明显
不是老化算法专用开关。因此当前最合适的中等置信度名称是
`battery_data_session_initialized`：表示这套电池遥测/数据会话已建立，允许消费者使用对应
路径。它不等同于 AC present、充放电方向、单纯 battery present，也不是“允许老化”的用户
设置。

当前实测 `0x45 = 0b01000101` 表示 bit 6、2、0 同时成立。只能确认 bit 0 对 `D1D1` 开放；
bit 2 和 bit 6 由其他电池状态机独立置位，bit 7 又有独立清除路径。按原始机器码的直接
读写，当前可以建立如下位级矩阵：

| 位 | 直接生产/清除证据 | 当前最稳妥解释 |
|---:|---|---|
| 0 | `0x4268` 写入 `0x81`；`0x427B` 整字节清零；多处消费者只测试 bit 0 | 数据会话/缓存初始化资格 |
| 1 | 未发现真实 `0x0497` 的直接读写 | 未使用或由未恢复的间接路径管理 |
| 2 | `0x564D`、`0x5D30` 以 `OR #04` 置位；未见 `AND #FB` 清除 | 电池遥测/发布阶段锁存位，正式语义未知 |
| 3–4 | 未发现真实 `0x0497` 的直接读写 | 未使用或间接管理 |
| 5 | `0x4D3B` 置位；`0x4327`、`0x4C2B` 清除 | 某个可重复进入/退出的电池处理阶段标志 |
| 6 | `0x5005`、`0x5085` 置位；未见直接清除 | 条件满足后的阶段完成/锁存位，正式语义未知 |
| 7 | `0x4268` 随 bit 0 一起置位；`0x2ACF`、`0x432F`、`0x5005` 等路径清除 | 初始化或电池状态机瞬态 busy 标志 |

因此没有证据把 `0x45` 整体解释为一个枚举或单一 aging 状态。尤其 bit 2/6 不能直接命名为
“充电允许”“电池存在”或“老化完成”；它们只是当前镜像中可观察到的阶段锁存行为。

在 `D1D1` 中，该位的精确作用是：signed current 为负后，若 `0x0497.0 == 0` 仍立即返回；
只有电池正在放电且数据会话资格成立，才继续检查 `0x0490.2`、处理应力并最终检查
`0x0490.0`。这使 stale-target 路径多了一道“遥测会话有效性”条件，但不改变充电稳态被
更早的 signed-current gate 拦截这一事实。

## 三条电压数据流

### 实时包电压不是 EC 算出来的

`CODE:0xE967` 从 `EC[0x0438:0x0439]` 取 16 位包电压。`D1D1` 只执行：

```c
high_voltage = battery_voltage_mv >= series_cells * 4100;
```

它不会把温度、循环次数或模式换算成实时包电压。实时值由外部 Fuel Gauge/Smart Battery
路径提供，EC 负责缓存和发布。

### 17600 mV 是电池侧基准

固件从电池遥测缓存块的偏移 `0x0E/0x0F` 取得这个 16 位字段；当前证据能确认
`XRAM[0x030F]` 是低字节、`XRAM[0x030E]` 是高字节，但不足以把 `0x0E/0x0F` 分别命名为
两条总线 byte 命令。当前组合值是：

```text
0x44C0 = 17600 mV = 4400 mV/cell for 4S
```

这不是 EC 中硬编码的设计电压，而是电池侧当前提供的未降额 charger 请求。

### 16600 mV 是保存目标

当 `D1D1` 完整走到尾部时，公式是：

```c
pack_derating_mv = series_cells * derating_per_cell_mv;
charge_voltage_target_mv = base_target_mv - pack_derating_mv;
```

当前两个值表面上满足：

```text
17600 - 16600 = 1000 mV
1000 / 4 cells = 250 mV/cell
```

但这个算术关系只说明保存值与当前基准相差一个 250 档，不证明它们是在同一次函数执行中
生成的。当前 gate 证据显示 `16600` 很可能是较早状态留下的目标。

## 完整充电电压限制路线图

对 `main-bank1.bin` 中全部直接 `MOV DPTR,#0x0522/#0x0523` 引用逐一检查后，可以把路线
分成六层。这里的“完整”指当前镜像中已经恢复的直接数据流；两个间接调度 gate 的正式
语义仍未恢复，不能伪装成已知状态名。

```text
电池遥测缓存 XRAM[0x030E:0x030F]（内部大端）
        |
        +--> 启动初始化 CODE:0x5A8E
        |       直接复制为 EC[0x0522:0x0523]（公开小端）
        |
        +--> 周期老化计算 CODE:0xD1D1（约 720 ms）
                gate -> 0/50/100/150/200/250 mV/cell
                target = base - cells * derating
                |
                v
        EC[0x0522:0x0523] desired target
                |
                +--> 正常路径：复制 desired
                |
                +--> 非正常路径：改用固定 0x3138（12600 mV）
                v
        XRAM[0x0836:0x0837] queued voltage
                |
                v
        XRAM[0x0832].1 request/retry -> XRAM[0x0833].1 active
                |
                v
        CODE:0x8EFB -> charger SMBus command 0x15
                |
                v
        外部 charger（本固件没有 ChargingVoltage readback）
```

### 第一层：基准字段的来源边界

`D1D1` 和初始化代码都把 `XRAM[0x030E:0x030F]` 当作 16 位基准使用，字节序为高、低；
初始化再交换成 `EC[0x0522]` 低字节、`EC[0x0523]` 高字节。当前 bank 能证明这个缓存字段
被消费，却没有恢复填充整个 `0x0300` 电池遥测块的底层事务，因此只能称其为电池遥测缓存
中的 charger voltage 基准，不能把偏移 `0x0E/0x0F` 当成两条 SBS 命令。

### 第二层：desired target 只有两个明确生产者

直接地址引用审计没有发现第三个 `0x0522:0x0523` 写入算法：

1. `CODE:0x5A8E initialize_battery_telemetry` 在初始化时直接复制基准；
2. `CODE:0xD1D1 update_battery_aging_charge_voltage_derating` 完整通过 gate 后写入降额结果。

其他看似相邻的引用有的是读取目标，有的是访问实际 `0x0524` 及之后的状态字段；不能受
`.d52` 的 `org -1` 符号后缀影响而误计为目标覆盖。这个结果意味着：若启动复制完成后
`D1D1` 一直提前返回，当前镜像中没有另一个直接维护者负责让 desired 跟随基准变化。

### 第三层：目标计算与 SOC 限制是两套策略

`D1D1` 只决定 charger 的 CV 目标，其输入是串数、循环次数、高压热应力、电池模式和模板
辅助量。Health/Balanced 模式还可能由其他状态机限制允许达到的 SOC；SOC 停充与这里的
`50–250 mV/cell` CV 降额可以同时生效，但前者不写 `0x0522:0x0523`，不能用来解释
16600 mV 的来源。

### 第四层：排队前存在正常值与固定回退值选择

`CODE:0x3196 is_charge_voltage_update_needed` 并非无条件比较 desired：

- 上游 gate 为真时，比较 `EC[0x0522:0x0523]` 与 `XRAM[0x0836:0x0837]`；
- gate 为假时，检查 queued 是否为固定 `0x3138`（12600 mV）。

`CODE:0x31FC queue_charge_voltage_update` 执行相同选择：真时复制 desired，假时把 queued
明确写成 12600，随后置位 `XRAM[0x0832].1`。因此 12600 是可实际发送的回退目标，不是
比较 sentinel。由于这两个函数通过间接调度使用，当前静态分析仍不能把 gate 精确命名为
“电池不存在”“遥测未就绪”或某个具体电源状态。

### 第五层：request、active 和 queued 是不同状态

`XRAM[0x0836:0x0837]` 保存即将发送的 word；`XRAM[0x0832].1` 是请求/重试位；worker
消费的是 `XRAM[0x0833].1` 活动位。三者不是 charger 寄存器。当前已恢复代码能确认请求
产生和活动消费，但 `0x0832 -> 0x0833` 的完整仲裁/提升路径跨共享调度代码，尚未恢复成
可以可靠命名的单一函数。因此路线图不能简化成“写 queued 后立刻发送”。

### 第六层：SMBus worker 不是成功闭环

`CODE:0x8EFB service_charger_smbus_queue` 消费活动 bit 1，构造 charger 地址 `0x12`
（标准 7 位地址 `0x09` 的 8 位写形式）和命令 `ChargingVoltage (0x15)`，再通过 common/
bank0 的 `CODE:0x80D2 execute_smbus_transaction` 执行事务。底层会记录 host status、超时和
`0xEE` 失败 sentinel，也能复位 SMBus 控制器。

但电压 worker 返回后没有直接依据传输成功标志完成闭环；它只是比较共享 word staging 与
queued mirror，不一致时重新置 `XRAM[0x0832].1`。这既不是 charger readback，也不能证明
所有 transport failure 都必然重试。要确认实际执行电压仍需总线抓取或 charger 专用读回。

### 路线图仍缺的两块

1. 电池遥测块 `0x0300` 的底层填充事务，以及基准何时从旧值刷新为 17600；
2. `3196/31FC` 的间接上游 gate 和 `0x0832 -> 0x0833` 仲裁路径的精确状态名。

这两块影响“为什么在某一时刻发布哪个目标”，但不改变已经确认的数值路径和两个 desired
写入者。启动 `base/desired/queued/request/active` 同步时间线是继续收敛它们的最低风险方法。

## `D1D1` 的关键控制流

### 第一处提前返回：清应力但不恢复目标

原始机器码 `CODE:0xD1E3–0xD1F6`：

```text
D1E3  LCALL E705          ; return XRAM[0x050B] & 0x80
D1E6  JNZ   D1F7
D1E8  MOV   DPTR,#09C7
D1EB  MOVX  @DPTR,A       ; A == 0
D1EC  MOV   DPTR,#09C8
D1EF  MOVX  @DPTR,A
D1F0  MOV   DPTR,#09C9
D1F3  MOVX  @DPTR,A
D1F4  INC   DPTR
D1F5  MOVX  @DPTR,A       ; clear 09CA
D1F6  RET
```

等价伪 C：

```c
if (!(XRAM[0x050B] & 0x80)) {
    XRAM[0x09C7] = 0;
    XRAM[0x09C8] = 0;
    XRAM[0x09C9] = 0;
    XRAM[0x09CA] = 0;
    return;                         // 0x0522:0x0523 untouched
}
```

这是应力清零后旧目标仍可保留的直接机制。代码从未在这个分支执行：

```c
charge_voltage_target = base_target;
```

也没有设置“目标无效/需要重算”标志。

### 第二处提前返回：放电时不写目标

通过方向 gate 后，函数会读取循环次数、串数、包电压和温度，并检查：

```c
if (!battery_data_session_initialized) // EC[0x0497].0
    return;
if (!(EC[0x0490] & 0x04))
    return;
```

应力处理之后，`CODE:0xD2D8` 又检查 `EC[0x0490].0`：

```c
if (!(EC[0x0490] & 0x01))
    return;
```

本次实测中 `0x0490.0` 与 AC 状态同步：断电为 0，接电为 1。因此本次正常放电虽然通过
`0x050B.7`，却在这里返回。机器码只能确认它是一个 gate；仅凭当前时间线还不能把它的
厂商正式语义确定为 AC present。

### 本次采样的稳态 gate 观察表

| 本机观察状态 | `0x050B.7` | `0x0490.0` | 是否到达目标写回 |
|---|---:|---:|:---:|
| 满电接 AC/零电流 | 0 | 1 | 否；清应力后在 `D1F6` 返回 |
| 正常充电 | 0 | 1 | 否；清应力后在 `D1F6` 返回 |
| 正常放电 | 1 | 0 | 否；在 `D2D8` 后返回 |
| AC 下电池仍为负电流 | 1 | 1 | 理论上可以 |

最后一行可能发生在极短接电过渡、适配器功率不足或 AC 下电池辅助供电时，但本次接电连续
采样没有观察到。即使存在短窗口，周期任务约每 720 ms 才调用一次，也不保证能命中。

### `D1D1` 内的目标写回尾部

完整通过 gate 后，`CODE:0xD3EC–0xD422` 才会：

1. 选择 `0/50/100/150/200/250 mV/cell` 档；
2. 乘以 `2/3/4` 串电芯数；
3. 从 `XRAM[0x030E:0x030F]` 基准做 16 位减法；
4. 以小端顺序写入 `EC[0x0522:0x0523]`。

本次观察到的普通充电和普通放电状态均到不了这段代码；这不证明所有平台和所有瞬态都不可达。

### `0x0522:0x0523` 为什么没有更新

启动初始化完成后，当前镜像只有 `D1D1` 会重新计算并写入 desired target。排队函数
`0x3196/0x31FC` 只读取 desired 或复制到 `0x0836:0x0837`，不会反向维护
`0x0522:0x0523`。按本机当前稳定状态逐项代入，阻塞点是确定的：

```text
充电：signed current >= 0
      -> 0x050B.7 == 0
      -> D1F6 清 stress 并 RET

放电：signed current < 0
      -> 通过第一 gate
      -> 0x0497.0 == 1（当前 0x45，此 gate 开放）
      -> 0x0490.2 == 1（当前 0x0E，此 gate 开放）
      -> 0x0490.0 == 0
      -> D421 返回，不写 desired
```

因此常规充电和常规放电分别被两个不同 gate 拦截。能够到达 `D3EC–D422` 写回段的关键组合
至少要求：

```text
signed_battery_current_ma < 0
&& (EC[0x0497] & 0x01)
&& (EC[0x0490] & 0x04)
&& (EC[0x0490] & 0x01)
```

在人类状态上，这对应“电源/AC gate 已成立，但电池仍在对系统放电”，例如插电瞬态、适配器
功率不足或混合供电。周期函数约 720 ms 才运行一次，短窗口未必会被命中。故当前
`0x0522:0x0523` 不更新不是 charger 队列丢包，而是 desired 的唯一周期生产者在常见稳态
控制流中不可达；16600 的最初来源仍可能是启动复制时的旧基准，或较早一次罕见窗口计算。

## 为什么重新刷写 EC 后仍是 16600 mV

刷写改变的是 SPI/EC firmware image，不等价于把所有运行期数据永久保持为零：

1. EC 重启后初始化代码会重新填充 XRAM；
2. Fuel Gauge 是电池侧独立器件，其遥测和学习值不因主 EC 刷写而清除；
3. `CODE:0x5A8E initialize_battery_telemetry` 内存在明确的基准到目标复制：

```c
EC[0x0522] = XRAM[0x030F];          // low byte
EC[0x0523] = XRAM[0x030E];          // high byte
```

4. 电池遥测是异步刷新的。初始化复制发生时的基准不一定等于稍后稳定读取的 17600 mV；
5. 如果初始化或较早状态写入 16600，而后来基准更新为 17600，`D1D1` 的提前返回会使二者
   长期失配。

第 3 点由机器码直接确认。第 4、5 点是一个优先验证的解释，但缺少启动早期
`0x030E/0x0522` 时间序列，也没有捕获实际生成 16600 的那次写入，因此不能据此确定根因。

## 当前 250 mV/cell 不能归因给什么

当前可见输入为：

| 变量 | 当前值 | 档位贡献 |
|---|---:|---:|
| 循环次数 | 57 | 0；低于首档 150 |
| 模式 | Normal | 0；Balanced/Health 才强制 100/200 档 |
| 老化路径辅助量 `0x0A54` | 固定为 1 | 0；低于首档门槛 7 |
| `0x0857` | RGB 模板路径 | 不参与老化函数 |
| 当前包电压 | 充电时约 16396 mV | 低于 4S 应力门槛 16400 mV |
| 当前应力 | `0x050B.7=0` 时周期路径会清零 | 执行清零路径后贡献 0；未同步直接读取应力字段 |

高压应力档位也不再是一个自然解释。common-bank `CODE:0x7B9E` 每个合格窗口实际只增加
`1/3/7`，不是 `0x100/0x300/0x700`；250 档要求应力严格大于 `0x3DE0`。即使一直处于
最高温度权重，也至少需要约 67.89 天合格高压时间，而且 `0x050B.7=0` 会把它清零。

因此，若假设当前电池和当前固件从未经历异常长的特殊状态，“旧基准/旧目标残留”是比
“当前应力真实触发 250 档”更值得优先验证的解释，但尚不能作为已确定根因。

## charger 队列为何不能证明实际执行电压

数据流末端为：

```text
EC[0x0522:0x0523] desired target
        |
        v
XRAM[0x0836:0x0837] queued mirror
        |
        v
Smart Battery Charger command 0x15 ChargingVoltage
```

`CODE:0x3196 is_charge_voltage_update_needed` 比较 desired 与 queued mirror；
`CODE:0x31FC queue_charge_voltage_update` 把 desired 复制到 queued mirror 并设置命令 bit；
`CODE:0x8EFB service_charger_smbus_queue` 发送 charger 命令 `0x15`。

但 `0x0836:0x0837` 是发送前复制的镜像，不是 charger readback。worker 也没有在本地路径中
读取 charger 的实际 CV 寄存器来确认执行值。因此：

- `16600 mV` 是固件期望发送的目标；
- `16396 mV` 是当时电池端实际电压；
- 二者不同并不矛盾，充电电流、终止条件、内阻、均衡和 charger 状态都会造成差值。

## 证据分级

### 机器码与采样直接支持

1. `0x0522:0x0523` 保存 16600，而当前 `0x030E:0x030F` 为 17600；
2. `0x050B.7=0` 分支清除应力后提前返回，不改目标；
3. 本次正常放电采样中 `0x0490.0=0`，该次调用无法到达目标写回；
4. 正常充电时实时 16.396 V 不是目标值，而是 `0x0438:0x0439` 遥测；
5. 当前循环、模式和辅助量不能即时产生 250 档；执行 `0x050B.7=0` 分支后，应力也不能贡献档位。

### 首要待验证假设

1. 16600 是初始化或较早状态留下的 stale target；
2. 后续基准变为 17600，而本机常见稳态 gate 组合阻止重新发布。

### 尚未证明

1. 初始化复制瞬间 `0x030E:0x030F` 的确切值；
2. charger 芯片最终接受并执行的 word 是否为 16600；
3. AC 下电池负电流状态是否能稳定触发 `D1D1` 尾部；
4. `0x050A:0x050B` 对应的上游遥测命令和四组内部电流副本各自的刷新条件；其 signed mA
   数值语义已经由机器码与同步采样确认。
5. 上述行为究竟是固件状态机遗漏、项目配置不匹配，还是只在尚未捕获的合法瞬态中完成更新。

## 建议的验证

### 启动早期时间序列

刷写或完整断电启动后，尽早以高频率只读：

```sh
sudo uv run tools/ec_rw.py read 0x030e 0x030f
sudo uv run tools/ec_rw.py read 0x0522 0x0523
sudo uv run tools/ec_rw.py read 0x050b
sudo uv run tools/ec_rw.py read 0x0490
```

目标是确认是否出现：

```text
boot:   base=16600, target=16600
later:  base=17600, target=16600
```

这会直接验证“异步基准刷新 + stale target”是否实际发生，但仍需结合写入点判断 16600 的来源。

### AC 下负电流验证

在不修改寄存器的前提下，观察高负载、适配器受限或接电切换时是否出现：

```text
0x050B.7 = 1 && 0x0490.0 = 1
```

若出现，再同步读取 `0x0522:0x0523`，可验证该窄状态是否真正触发尾部写回。

### charger 总线验证

要证明外部 charger 实际接收值，需要 SMBus 抓取或增加固件侧事务日志。仅读取 queued
mirror 不足以作为硬件确认。

## 固件修复方向

### 方案 A：清应力后继续完整重算

最小实验思路是让 `CODE:0xD1F6` 不再 `RET`，而是继续执行 `D1F7` 后的输入收集和档位
决策。当前指令是单字节 `RET (0x22)`，从布局上改为 `NOP (0x00)` 会顺序落入 `D1F7`。

优点：

- 清零应力后仍会尊重循环次数、Balanced/Health 模式和电芯数；
- 在本机观察到的普通充电稳态可以重新生成目标；
- 当前输入下反事实结果应为 17600 mV。

风险：

- 这会改变厂商原本用 `0x050B.7` gate 整个算法的语义，而不仅是修复缓存；
- 17600 mV 对 4S 等于 4.4 V/cell，必须确认电芯和 charger 设计允许；
- 函数调用频率约 720 ms，可能造成此前未预期的重复目标更新；
- 必须重新处理镜像校验、bank 映射和恢复方案。

因此该单字节变化只能作为实验假设，不能直接称为安全修复。

### 方案 B：增加 dirty/recompute 路径

更稳妥的工程修复是：

1. 在基准 `0x030E:0x030F` 变化时设置 dirty flag；
2. 允许目标选择逻辑在应力为零时运行，但保留循环次数和模式降额；
3. 仅在新目标与 `0x0522:0x0523` 不同时写回；
4. 通过既有 `0x3196/0x31FC` 队列路径发送 charger 命令；
5. 增加成功/失败日志或 readback，而不是只依赖 queued mirror。

这需要 code cave 或较大补丁，但语义比简单移除 gate 清晰。

### 不推荐：直接复制 17600 或清零目标

在 `0x050B.7=0` 分支直接执行 `target=base` 会绕过循环次数和 Health/Balanced 降额；直接把
目标写成零则可能触发 charger 禁充或异常回退。二者都不应作为正式修复。

主机也无法用 `ec_rw.py` 直接修正这些字段：`0x050B` 和 `0x0522:0x0523` 位于
H2RAM `0x500–0x5FF` 写保护区，主机写事务会被硬件忽略。

## 最终判断

当前现象不是“EC 刷写后 XRAM 没有清空”这么简单。代码明确存在清除应力但不恢复
`EC[0x0522:0x0523]` 的路径；本机本次放电和充电稳态采样中的 gate 组合又分别阻断尾部，
因此 16600 mV 具备长期保留的控制流条件。现有证据尚未捕获生成 16600 的那次写入，所以
这里只能确认“旧目标得以保留的机制”，不能确认“旧目标的来源及最终根因”。

最值得优先验证的是启动早期 `base/target` 时间序列。若确认基准从 16600 变为 17600 而
目标不变，就可以把根因进一步收敛为“异步基准刷新后缺少目标失效/重算机制”。
