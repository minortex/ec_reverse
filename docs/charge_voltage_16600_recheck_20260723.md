# 16600 mV 充电目标重新归因（2026-07-23）

## 结论

当前 `16600 mV` 的直接计算原因已经闭合，但不能把它无条件写成“INT1 导致”：

```text
base = 17600 mV
cells = 4
auxiliary = 0x1B = 27 >= 25
per-cell derating = 250 mV
target = 17600 - 4 * 250 = 16600 mV
```

2026-07-23 的只读实机快照同时排除了其它正常分档输入：循环次数为 60、模式为 Normal、
high-voltage stress 为 `0x003C`，三者都不足以产生任何降额。周期 worker 每推进一次
`0x09C7`，都能观察到 `0x0A54=0x1B` 窗口。因此，固件选择 16600 而非 17600 的直接原因
是异常 auxiliary 值命中最高档。

INT1 ISR 与主线共用 bank2 且不保存 `R7` 是官方固件中确定存在的 ABI 缺陷，也是当前找到
的唯一异步 `R7` 改写候选。INT1-R7 补丁后 17600 保持稳定的 A/B 结果进一步支持它参与
故障，但现有观测仍没有对“产生每一个 `0x1B` 的那次 INT1 入口”计数或单步。因此应表述为：

- `0x1B` 是 16600 的直接 producer：已确认；
- INT1 能污染该 `R7`，且补丁能消除现象：强支持；
- 每次 `0x1B` 均由 INT1 动态产生：尚未最终证明。

## 官方固件二进制链

蓝本为 `samples/GXxHXxx_21.200`，大小 256 KiB，SHA-256：

```text
34c050d30772da07ef262fc7016e0677b9b1b4cdcd90cf43d93f0f15bf6a38c2
```

以下地址均由原始 opcode 解码。`D1D1` 位于主 EC bank1；`0x1D08` 位于 fixed common 区。

```text
bank1:D250  12 D1 BB       LCALL bank1:D1BB
bank1:D253  EF             MOV A,R7
bank1:D254  70 03          JNZ D259
bank1:D259  12 E1 A0       LCALL bank1:E1A0
bank1:D25C  70 03          JNZ D261
bank1:D261  12 1D 08       LCALL common:1D08
bank1:D264  90 0A 54 EF F0 MOV DPTR,#0A54; MOV A,R7; MOVX @DPTR,A
bank1:D269  90 03 9D F0    MOV DPTR,#039D; MOVX @DPTR,A
```

`bank1:D1BB` 的成功路径以 `7F 01 22` 返回 `R7=1`。`bank1:E1A0` 的原始字节为
`90 04 90 E0 54 04 22`，只读取 `EC[0x0490].bit2`，不写 R7。

`common:1D08` 的原始字节为：

```text
90 E6 4F 02 11 14
```

即把目标 `bank1:E64F` 装入 DPTR 后跳到 bank1 thunk `common:1114`。目标原始字节为：

```text
bank1:E64F  90 08 E2 E0 44 20 F0 90 09 83 E0 54 10 22
```

它设置 `XRAM[0x08E2].bit5`，读取 `XRAM[0x0983].bit4` 并返回，不直接写 R7。thunk 保存
bank selector、ACC 和 DPTR，也未发现写 R7。因此在没有异步打断时，D264 应写入 1。

最终分档从最高档向下判断。auxiliary 的最高档比较位于 `bank1:D31A` 一带；命中后把
`R3=250`。尾部先计算 `cells * R3`，再从 `0x030E:0x030F` 的基准中作 16 位减法，原始
写回指令为：

```text
bank1:D406  90 05 23 F0   MOV DPTR,#0523; MOVX @DPTR,A  ; target high
bank1:D41E  90 05 22 F0   MOV DPTR,#0522; MOVX @DPTR,A  ; target low
```

所以 `0x0522:0x0523` 是最终 little-endian desired target，而不是电池提供的原始基准。

## 当前寄存器是否满足

2026-07-23 01:10 的新一轮 50 ms 只读采样推翻了“当前 gate 未满足”的解释。AC 在线时：

| 输入 | 实测值 | 结论 |
|---|---:|---|
| `0x0490` | `0x0F` | bit0/1/2 三个 gate 全部开放 |
| `0x0497` | `0x45` | 电池数据会话 gate 开放 |
| `0x0741` | `0x81` | AP 存在 |
| base `0x030E:0x030F` | `0x44C0` = 17600 | 未降额基准正常 |
| target/queued | `0x40D8` = 16600 | desired 和队列一致 |
| cycles/mode/stress | 60 / Normal / `0x003C` | 均不足以降额 |
| `0x0A54` | 稳态 `0x02`，每秒短暂 `0x1B` | `0x1B` 与 `0x09C7` 增量同步 |
| `0x039D` | `0x1B` | 锁定 D264 当次写入的 R7 |

因此不能用单次读到的稳态 `0x0A54=2` 判断分档条件未满足。真正被 `D1D1` 消费的是短暂的
`0x1B` 窗口，它持续时间约一个 50 ms 采样点并在每次周期写回时出现。

2026-07-23 00:56（Asia/Shanghai）使用官方仓库的只读 I2EC monitor 采样 12 秒：

| 输入 | 当前值 | 是否能选择 250 mV/cell |
|---|---:|---|
| base `0x030E:0x030F` | 17600 mV | 只是减法基准 |
| cells `0x0491.7:6` | 4S | 只决定乘数 |
| cycles `0x04A6:0x04A7` | 60 | 否，最高档需 `>=550` |
| mode `0x07A6.5:4` | Normal (`0x00`) | 否 |
| stress `0x09C9:0x09CA` | `0x003C` | 否，第一档也需 `>0x10E0` |
| auxiliary `0x0A54` | 周期出现 `0x1B` | 是，最高档需 `>=25` |
| desired `0x0522:0x0523` | 16600 mV | 与公式一致 |
| queued `0x0836:0x0837` | 16600 mV | 已复制低目标 |

采样还观察到 `0x0A54` 在 worker 周期附近按 `0x2F/0x2E -> 0x1B -> 0x02` 变化，而
`0x09C7` 同步递增。这里的 `0x0A54` 是共享 scratch，只有与 D264 的唯一影子
`0x039D` 结合才能把 `0x1B` 锁定为该次 worker 写入的 R7；此前现场已经观察到
`0x0A54=0x039D=0x1B`。

## 为什么当前看起来始终是 16600

早先离电采样时 `EC[0x0490]=0x0E`，当时确实只会保留旧目标；但 01:10 的 AC 在线采样为
`0x0F`，所有 gate 均开放，不能再把当前 16600 解释成单纯旧值锁存。

因此当前状态应准确解释为：

1. worker 周期性生成异常 `0x1B`；
2. AC 在线时完整执行最高档减法并重新写入 16600；
3. 手工把 `0x0522:0x0523` 写为 17600 后，不到约 1 秒即恢复为 16600，实验证实周期重写；
4. AC 断开、gate0 为 0 时，已有 16600 才转为旧值保留。

AC 在线且 gate0/1/2 均开放时，下一次完整 D1D1 会消费 `0x1B`，重新算出并写入 16600。
手工写 17600 只能维持到下一次完整周期，原因是 D1D1 每次从 17600 基准重新执行相同的
最高档减法。物理 block1 的 `0x15C5D/0x15C65` 虽包含形似目标写入的字节，但逻辑 `0x5Cxx`
属于 fixed-common 区，运行时映射 block0；它是 inactive low-window 数据，不是第二个可执行
writer。按有效 bank 映射穷举后，D1D1 仍是目标的直接 writer。

## INT1 归因边界

主 EC 的非空 ISR 中，Timer0/Timer1 虽也选择 bank2 且不保存 R0--R7，但其 ISR 内调用的
`common:0EAC/0ECB/0EC0` 原始代码只操作 timer SFR 和标志，不写 R7。INT1 则会进入邮箱
分派，明确使用 R6/R7，并且入口不保存它们。因此目前二进制候选仍只剩 INT1。

不过 `0x1110` 在此前 95 秒追踪中始终为 `0x10`，没有观测到与每次 `0x1B` 对应的命令变化；
ISR 又可能经 `2D.6=1 -> common:14C8` 绕过 `0x1110`。仅靠 XRAM 轮询不能区分这两种入口，
也不能给 INT1 事件计数。INT1-R7 补丁改变了 ISR 字节和少量执行时序，所以 A/B 是强证据，
但还不是逐次因果观测。

要最终闭环，应在官方固件副本上增加只递增的 INT1 入口计数和 D264 前 R7 捕获，分别覆盖
`2D.6` 两条分支；或者使用 ITE 调试接口在 D261--D269 窗口单步。未获得这项证据前，不应
把“INT1 风险已确认”扩大写成“16600 的每次动态触发均已确认来自 INT1”。
