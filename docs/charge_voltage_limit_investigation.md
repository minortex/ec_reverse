# 充电电压限制调查：16.6/17.0 V、冷启动与 AC 重算

## 当前结论

本机至少存在三条不同的电压状态，不能把它们当成同一个“电池电压限制”：

| 状态 | 地址 | 本次确认 |
|---|---|---:|
| 实时电池包电压 | `EC[0x0438:0x0439]` | 约 `16818 mV` |
| 未降额基准 | `EC[0x030E:0x030F]` | `17600 mV` |
| EC 保存的充电目标 | `EC[0x0522:0x0523]` | `16600` 或 `17000 mV` |

`0x0522` 是低字节、`0x0523` 是高字节；例如原始 `0xC044` 表示 `0x44C0 = 17600 mV`，
原始 `0x6842` 表示 `0x4268 = 17000 mV`。

最终目标由主 EC 的老化补偿函数 `CODE:0xD1D1` 周期性重算：

```text
pack_derating = series_cells * per_cell_derating
charge_target = base_target - pack_derating
```

当前为 4S，因此 `17600 -> 17000` 正好是 `600 mV = 4 * 150 mV/cell`。

## 现象时间线

### 1. 初始 16.6 V

在早期接电状态曾观察到：

```text
base   = 17600 mV
target = 16600 mV
mode   = 0x00 (Normal)
cycle  = 57
```

随后重新刷写 EC，刷写过程必然触发 EC reset，但目标仍回到 `16600 mV`。这说明：

- `16600` 不是简单的当前温度读数；
- 单纯刷写 EC 不能证明 aging state 被清零；
- reset 后的 AC gate 和 XRAM 初值顺序会影响首次目标计算。

### 2. 断开 AC、由电池供电

此前已经完成一次无电池启动和电池重新连接；随后断开 AC、只由电池供电时读到：

```text
0490 = 0x06
030E/0F = 17600 mV
0522/23 = 17600 mV (raw 0xC044)
07A6 = 0x00
04A6/4A7 = 57
04A2/4A3 = 0x0BD6 (约 30.3°C)
```

这一步证明不需要低温，目标就可以在电池供电阶段显示为未降额基准。此时
`0x0490.bit0` 未开放，目标没有被最终 charger 更新路径重算；所以这个 `17600` 本身还
不能证明 aging 档位已经被清零。

### 3. 重新插入 AC

只读监听捕获到完整过渡：

```text
插入前       0490=0x06  target=17600
AC 过渡      0490=0x07  target=17600
约 1 秒后    0490=0x0F  target=17000
随后稳定     0490=0x0F  target=17000
```

基准始终保持 `17600 mV`，变化只发生在 `0x0522:0x0523`。因此 AC 重新上线后，
`D1D1` 在三个 power-source gate 全开放时重新计算出 `150 mV/cell` 档。

## 对“冰箱是否必要”的判断

冰箱不是必要变量。冰箱实验同时改变了温度、电池会话和电源状态，不能把结果归因于温度；
当前时间线已经证明，关键动作是：

1. 电池会话/启动顺序使目标暂时显示为 `17600`；
2. AC 上线使 `0490.bit0` 开放；
3. 主 EC 随后重新消费 aging state 并写回 `17000`。

约 30°C 的温度只决定 high-voltage stress 每个合格窗口增加 `+3`，不会在一次温度变化
时直接把目标改成 16.8 或 17.0 V。因而“放进冰箱”更可能只是帮助完成一次明确的电池会话/冷启动，
不是触发降压的物理条件。

## 为什么 Normal 仍是 17.0 V

`EC[0x07A6].5:4` 只是模式最低档，不是清除全部老化状态的开关。固件等价于选择多个来源
中的最高档：

```text
effective_tier = max(stress_tier, cycle_tier, auxiliary_tier, mode_floor)
Normal   floor =   0 mV/cell
Balanced floor = 100 mV/cell
Health   floor = 200 mV/cell
```

已知档位门槛如下：

| 每电芯降额 | 循环次数 | high-voltage stress | 模式/辅助条件 |
|---:|---:|---:|---|
| 50 | >=150 | > `0x10E0` | 辅助等级 >=7 |
| 100 | >=250 | > `0x1950` | Balanced 至少触发 |
| 150 | >=350 | > `0x21C0` | 辅助等级 >=13 |
| 200 | >=450 | > `0x2D00` | Health 至少触发 |
| 250 | >=550 | > `0x3DE0` | 辅助等级 >=25 |

当前循环次数只有 57，Normal 又没有模式下限，辅助路径在本机局部到达值为 1；所以
`17000` 不能由这三项解释，剩余候选是隐藏 high-voltage stress 或未命名的持久/共享状态。

高压 stress 的形成条件是包电压达到 `4 * 4100 = 16400 mV`，由约 720 ms 任务累计：

| 温度 | 每个合格窗口增量 |
|---|---:|
| `<29.85°C` | `+1` |
| `29.85–39.85°C` | `+3` |
| `>39.85°C` | `+7` |

不过复位审计发现 `XRAM[0x09C7:0x09CA]` 没有确定的 C51 启动清零覆盖。完全掉电并不等于
aging word 已被置零；若首次 `D1D1` 执行时 `0490.bit1` 已为 1，固件可能直接消费未初始化
SRAM 内容。这可以解释同一机器在不同 reset/刷写顺序下出现 `16600` 与 `17000`。

## gate 与数据流

- `0490.bit1`：aging stress 清零/累计 gate；为 0 时清 `0x09C7–0x09CA` 并返回。
- `0490.bit2`：辅助 hook gate。
- `0490.bit0`：最终目标写回/AC 相关 gate；关闭时旧 target 可以保留。
- `0x05B9 == 1`：老化函数提前返回。
- `0x030E:0x030F`：电池侧提供的未降额基准，不是固定设计值。
- `0x0522:0x0523`：EC 选出的 desired target，之后复制到 queued mirror 并排队发送
  Smart Battery Charger `ChargingVoltage (0x15)`；当前没有 charger readback，不能据此证明
  外部 charger 实际执行值。

PD bank0/bank1 的直接地址审计没有发现 `0x07A6`、`0x0522` 或 aging accumulator 的
写入/计算引用。PD 可能间接影响 USB-C 输入功率，但本次 `17600 - 4*150 = 17000` 的
精确结果由主 EC aging 路径即可解释。

## `0x0490` 的离线写入生产者

这次对原始 `main-bank1.bin` 中 `MOV DPTR,#0x0490` 的全部引用进行了逐项分类；`.d52`
中的 `dptr_048F` 已按 `org 0-1h` 偏差还原为真实地址 `0x0490`。main-bank0 只有六处
直接读取，PD bank 未发现相关路径；全部直接写入均在 main-bank1。

| 代码位置 | 操作 | 直接效果 | 角色判断 |
|---:|---|---|---|
| `0x1471` | `A &= 0xFE` | 清 bit0 | 电池/电源状态 teardown |
| `0x1518` | `A |= 0x01; A &= 0x77` | 置 bit0，同时清 bit3/bit7 | 状态恢复路径 |
| `0x3B32` | `A |= 0x02; A &= 0xEB` | 明确置 bit1；随后清 bit2/bit4 | 唯一确认的固件 bit1 置位函数 |
| `0x3C69` | `A &= 0xF9` | 清 bit1/bit2 | 电池状态错误/超时路径 |
| `0x3C69` 邻接基本块 | `A |= 0x20; A &= 0xBF` | 置 bit5、清 bit6 | 独立电源状态位 |
| `0x3D4A` | `A |= 0x08; A &= 0x7F` | 置 bit3、清 bit7 | 状态转换路径 |
| `0x3D5D` | `A |= 0x80; A &= 0xF7` | 置 bit7、清 bit3 | 状态转换路径 |
| `0x3D70` / `0x3DD9` | `A &= 0x77` | 清 bit3/bit7 | 电池遥测/会话收尾路径 |
| `0x5D30` | `A |= 0x04` | 置 bit2 | 电池数据初始化完成路径 |

上述地址是以原始 bank1 二进制重新建立函数入口后的地址；旧 `.d52` 中的 `jump_3B6A`、
`jump_3C72` 等只是函数内部基本块标签，不能当作函数入口。除主机通过 H2RAM 写入外，
静态代码没有发现另一个直接置 bit1 的函数。

因此手工写入 `0x0D` 后出现 `0x07` 是可以由固件写入链解释的，而不是 EC 随机恢复：

```text
主机写入       0x0D  (bit1=0)
3B32           置 bit1，并清 bit2/bit4
5D30           置 bit2（若该初始化路径执行）
3D70/3DD9      清 bit3/bit7
最终           0x07
```

`0x3B32` 并不是一个无条件刷新函数：它先检查 `XRAM[0x05F3].1`，再通过
`CODE:0xC078(0x05F2)` 判断倒计时/状态是否到期；到期时才置 bit1。对应的 `0x3C69` 会先
把 `0x05F2` 设为 3，在 bit1 已置位且 `CODE:0xC078(0x05F3)` 到期时清 bit1/bit2。
这组 `0x05F1–0x05F4` 计数器解释了为什么主机一次写入后 bit1 的保持时间不确定，也解释了
为什么拔插会话会重新建立 `0x0490` 的组合状态。

### `0x05F1–0x05F5` 的来源与拔插状态转换

原始 main-bank0 ROM 表 `0x6BC0` 包含三字节初始化项 `address_hi, address_lo, value`：

```text
06 C0 14
05 F2 05
05 F4 05
05 F3 05
05 F5 05
06 DD 05
```

因此 `0x05F2–0x05F5` 的确定初始化值均为 5；它们不是未初始化 aging SRAM。各字段的数据
流为：

| 字段 | 装载 | 消费 | 作用 |
|---:|---|---|---|
| `0x05F1` | `CODE:0x14E8` 写 `200` 或 `1` | `CODE:0x1444` 递减 | `0490.bit0` 的 AC 资格去抖 |
| `0x05F2` | ROM 初值 5；`0x3C69` 重装 3 | `0x3B32` 递减 | 电池会话开始、bit1 置位去抖 |
| `0x05F3` | ROM 初值 5 | `0x3C69` 递减 | 电池会话结束、bit1/bit2 清除去抖 |
| `0x05F4` | ROM 初值 5 | `0x3BB3` 递减 | power-source bit5 置位去抖 |
| `0x05F5` | ROM 初值 5 | `0x3CE0` 递减 | power-source bit5/bit6 清除去抖 |

`CODE:0x14E8` 在 bit0 尚未建立时等待计数器到期，随后执行
`0490 = (0490 & 0x77) | 0x01` 并重建 charger/AC 状态；`CODE:0x1444` 在相反事件上递减
`0x05F1`，到期后清 bit0。由实测 AC 插入 `0x06 -> 0x07 -> 0x0F` 可知，bit0 是 AC 相关
资格位，但 bit3 还来自另一条状态路径。

电池拔插对应另一组序列：

```text
电池会话开始：3B32 置 bit1，清 bit2，清 0x0497
遥测初始化：  5A8E 刷新缓存与公开字段
会话就绪：    5D30 置 bit2，0x0497 随初始化路径重新建立
电池会话结束：3C69 清 bit1/bit2，并执行遥测 teardown
```

因此 bit1 更准确的含义是“电池会话存在/允许”，bit2 是“电池遥测初始化完成/辅助路径
允许”，而不是两个独立的 AC 位。拔插之所以能恢复 `0x0490`，是因为它重新执行了这条完整
会话序列；主机单写 `0x0490` 不会同步重装 `0x05F2/0x05F3` 或重建 `0x0497`。

2026-07-19 的只读实机对照进一步吻合该模型：AC 稳态 `0490=0x0F`、`0497=0x45` 时，
`05F1=C8 (200)`、`05F2=01`、`05F3=03`、`05F4=03`、`05F5=05`。这些值与 AC 资格计时器
和电池会话去抖已完成/正在收尾的状态相符，确认它们是运行期状态字段而非反汇编假象。

### 无电池启动的自然清零窗口

2026-07-19 在拆下电池、仅接适配器启动后，以 50 ms 间隔纯只读监听：

```text
0490 = 0x01       bit1=0, bit2=0, bit0=1
0497 = 0x04       电池会话未初始化
0741 = 0x01       ApExistFlag=1
05B9 = 0x00       D1BB gate 开放
05F1-F5 = C8/03/03/03/05
target = 0        无电池时为回退/无效值，不用于电压判断
```

`0490.bit1` 从采样开始持续为 0，达到 `755 ms` 时触发监听器的
`CLEAR-WINDOW>=720ms` 标记。这个标记只确认 bit1 低电平在墙钟时间上覆盖了一个正常
`D1D1` 周期，不能单独证明无电池状态下 `CODE:0x8FFB` 仍会调用 `D1D1`；无电池时
电池维护路径可能被跳过。

重新装回电池并接入 AC 后，当前只读稳态为：

```text
0490 = 07   0497 = 45   05F1-F5 = C8/01/03/03/05
0741 = 81   05B9 = 00   0522:0523 = 17000 mV
0836:0837 = FFFF   030E:030F = 17600 mV
```

电池会话已经重建，D1D1 的 bit1/bit2/bit0 与 `05B9` gate 均开放，但目标仍为 17000 mV。
因此无电池窗口没有产生可见的 stress 清零结果；仍需区分“无电池跳过 8FFB→D1D1”和
“会话建立后又恢复/形成 150 档”两种解释。

### 2026-07-19 AC 拔插监听

`/tmp/log.txt` 的 50 ms 采样显示：

```text
11:44:13  0490=07  05F1=C8  target=17000
11:44:15  05F1=BE ...
11:44:16  05F1=01
11:44:16  0490=06  target=17000
11:44:35  05F1=C8
11:44:35  0490=07
11:44:35  0490=0F  target=17000
```

AC 拔出后只有 `0490.bit0` 清除，`bit1` 始终为 1，`0497` 始终为 `45`，`05F2/05F3`
也没有变化；AC 插回后 bit0 按 `06 -> 07 -> 0F` 重新建立。整个过程中 `0522:0523`
始终为 `17000 mV`。因此普通 AC 拔插不会触发电池会话重建，也不会触发 D1D1 的 bit1=0
清零分支；17 V 限制与 AC 去抖状态独立存在。

上述基本块的确切调用顺序仍受多个电池状态机分支控制，但它足以说明 `0x0490` 是多个
生产者共享的状态字，不是一个可独立保持的 aging 控制位。你这次 `0x0D -> 0x07` 的现象
与“bit1 在一次 720 ms 调度到达前被重新置回”一致。

## `D1D1` 调用前后的完整 gate 顺序

### 调度层复核（离线）

Common-bank Timer0 ISR 只负责重装定时器并置 `26h.5`。主循环的 `0x0D3F` worker 检查该
标志后无条件调用 `0x0DCA`；`0x0DCA` 先执行 `0x3B63`、`0x0E70`，再按 `0x44/0x45`
相位计数器通过内联跳转表选择任务。表项 `0x0EA3 -> 0x1588 -> 0x8FFB`，该调度路径在
进入跳板前没有电池存在位或 `0497` 判断。因此无电池启动时不能简单假设整个 720 ms
worker 被跳过。

需要注意：`0x8FFB` 是 banked 地址。在完整固件反汇编中它位于一个上层维护函数内部，不能
仅凭地址名称断言它等同于 `D1D1`；还需继续解析其 `0x0A4C–0x0A50` 状态 gate 以及 bank
切换/间接调用，才能证明无电池时是否最终到达 `CODE:0xD1D1`。

`CODE:0x8FFB periodic_720ms_power_maintenance` 每约 720 ms 调用一次 `D1D1`；调用者在
它之前只执行通用维护函数，没有额外的 aging 条件。`D1D1` 内部顺序为：

1. 检查 `EC[0x0741].0`（ApExistFlag）；为 0 时只清 `0x07A6` 的模式位，继续执行。
2. 调用 `CODE:0xE705` 读取 `EC[0x0490] & 0x02`。
3. 若 bit1 为 0，立即清零 `0x09C7–0x09CA` 并 `RET`。此分支不检查 `0x05B9`、bit2、
   bit0，也不写回 `0x0522:0x0523`。
4. bit1 为 1 时，解码串数、读取循环次数/温度，并调用 `CODE:0xD1BB` 检查
   `0x05B9 != 1`；失败则返回。
5. 检查 `EC[0x0490].2`；为 0 则返回，不清 stress。
6. 执行高压计数、温度权重和辅助量计算。
7. 调用 `CODE:0xDF87` 检查 `EC[0x0490].0`；为 0 则返回，不写目标。
8. 计算最高老化档，并将结果写入 `EC[0x0522:0x0523]`。

所以本次写入没有改变目标，最直接的解释不是后续 gate 拦截，而是 `0x0D` 窗口没有覆盖
下一次 `0x8FFB -> D1D1 -> E705` 调用；在调用前 bit1 已被 `0x3B32` 等状态路径恢复。
这也解释了为什么手工写入后读到 `0x07`，而 `0522:0523` 仍保持 `17000 mV`。

随后以 100 ms 间隔连续监听超过一分钟，`0x0490` 始终为 `0x07`，target 始终为
`17000 mV`。同步补采为：

```text
0x0490 = 0x07   (bit2/bit1/bit0 全为 1)
0x05B9 = 0x00   (D1BB gate 开放)
0x0741 = 0x81   (ApExistFlag bit0 为 1)
0x0497 = 0x45   (电池数据会话已初始化)
```

因此 `0x07` 稳态本身不会阻止 `D1D1` 计算。若先前的 `0x0D` 真被某次 `D1D1` 采到，清零
后下一次开放周期应在循环 57、Normal 模式下计算出 `17600 mV`；长期仍为 `17000` 表明
清零分支大概率没有执行。主机单次写入的 bit1=0 窗口短于 720 ms 调度间隔，是与全部实测
相容的解释。

后续观察表明，`0x07` 不会自行回到 `0x0F`；必须经过电源或电池会话的拔插/重建，
`0x0490` 才恢复完整状态。这说明 bit3 等字段是会话状态机的锁存结果，而不是
`D1D1` 周期任务每次都会重建的值。它也解释了为什么单次主机写 `0x0D` 既没有稳定保持，
也没有把目标带回 `17600 mV`。
