# EC 电池校准逻辑调查（2026-07-21）

## 本文已被重定位审计取代

主 EC 实际使用固定 32 KiB common 加三个 32 KiB bank window；本文最初把
`main-bank1.bin` 低 32 KiB 文件偏移直接当成逻辑 CODE 地址，后来的“低半区不可达”勘误
也因此不成立。文件偏移 `0x47EA/0x4970/0x5A8E/0x69F6` 应分别重定位为逻辑
`0xC7EA/0xC970/0xDA8E/0xE9F6`。

最新静态分析已经确认 `RSOC <= 12`、低电压起点、充电累计、100% 后 ChargingCurrent
归零、64 次终点确认、单轮约 +/-300 mAh 限幅和 FCC 重新发布。请以
[`battery_calibration_trigger_audit_20260721.md`](battery_calibration_trigger_audit_20260721.md)
为准。以下内容保留为调查历史，其中“不可达”及 `>=12`、高于 14820 mV 等结论均已作废。

## 旧版 bank 有效性审计（已作废）

此前本文件把 `main-bank1.bin` 低半区的若干连续字节当成可执行函数，这是不可靠的。
当前 bank-wrapper 提取结果显示，bank1 的有效 far-call 目标集中在 `0x8000` 以上；
`0x4433`、`0x4970`、`0x5A8E`、`0x69F6` 这些地址不能仅凭 `.d52` 连续反汇编证明是
当前 CPU 会执行的 bank1 入口。`0x47EA` 附近首先还是一个跳往 `0xC80C` 等地址的
switch 表，表后代码没有有效调用交叉引用。

因此本文后续关于“0x03A3 七状态校正机”“0x0388/0x038A 直接驱动 FCC”“0x5A8E
写回 0x0404”的结论全部降级为**待撤回假设**。重新审计时，在有效 bank 代码中目前只
确认了 `0x0404:0x0405` 的读取，没有找到直接写入它的可执行指令。当前不能排除 FCC
变化来自 Fuel Gauge、另一条有效 bank 路径或 ACPI/主机同步层。后续结论必须以有效
wrapper/控制流和真机变化顺序重新建立。

### 有效代码中的“12%”关系

仅检查 fixed/common 和 bank1 `0x8000–0xFFFF` 后，没有找到“RSOC < 12 时启动 FCC 校正”
的可执行路径。最接近该说法的有效 helper 是 bank1 `CODE:0xE1CA`：

```text
90 04 AB       MOV  DPTR,#0x04AB
E0             MOVX A,@DPTR
D3             SETB C
94 0A          SUBB A,#0x0A
74 80 94 80    signed-compare tail
22             RET
```

由于 `SETB C` 带入一次 borrow，该比较边界相当于 `RSOC <= 10`，口语可被描述为“低于
11%/12%”。它的有效调用者位于 `0xB3xx`、`0xC7xx` 平台风扇/功耗状态路径，结果控制
`XRAM[0x1807]` 等 fan/PWM 派生状态；没有控制流连接到 `EC[0x0404:0x0405]` FCC 写入。

其他已确认的有效 RSOC 门槛包括约 3%、35%、41%、50%、71% 和 100%，分别服务于保护、
平台策略和模式切换。此前低半区 `0x0514` 与 `0x0C` 的比较没有有效入口，不能用于证明
校正规则。若电池必须放到 12% 以下才学习 FCC，该规则更可能位于电池包 Fuel Gauge 内部，
而不在当前已确认的主 EC 可执行代码中。

## 结论

在 `samples/disasm/main-bank1.bin` 中找到了 **EC 侧的容量统计、满电确认、比例校正和发布
状态机**。它会在一次符合条件的电池会话中积累内部候选量，在满电条件稳定后更新比例因子，
再重算并发布操作系统看到的 FullChargeCapacity。此前仅分析 `0x4970/0x5A8E` 出口而写下的
“没有检测完整周期”结论过强，现予以撤回。

仍未发现 EC 向 Fuel Gauge 发送 ManufacturerAccess 学习/复位命令；因此这里可以确认的是
**EC 自己维护并发布的容量校正**，不等于已经证明 EC 改写了 Fuel Gauge 内部 learned FCC。

当前最稳妥的职责划分是：

- 外部 Fuel Gauge 提供电流、电压、电量、RSOC 和循环次数等基础遥测；
- EC 在 `XRAM[0x0361...]`、`0x0384...`、`0x0388...` 一带维护校正会话和候选统计；
- 满电条件持续成立后，EC 生成一个受上次结果动态上下界约束的比例因子；
- EC 用该因子缩放并发布给 ACPI/操作系统的 LastFullCapacity；
- 因而用户观察到的 `3700 -> 4000 / 5200` 可以由这条 EC 路径直接解释。

## 校正状态机

`CODE:0x47EA` 读取 `XRAM[0x03A3] & 7`，原始跳转表明确分派 0 至 6 七个阶段。以下地址
均按二进制中的真实 DPTR 立即数书写；旧 `.d52` 符号普遍偏小 1，不能照抄标签名。

| 状态 | 入口 | 已确认行为 |
|---:|---:|---|
| 0 | `0x480C` | 要求 queued ChargingCurrent `0x0834:0x0835 != 0`、`XRAM[0x0514] >= 12`、`0x03A2.bit3`，并处于高电压起点；建立本轮相对上次 FCC 的允许区间 |
| 1 | `0x486D` | 比较 pack voltage `0x0384:0x0385` 与低电压阈值；正常高电压入口会清充入容量累计和会话计数，准备统计尾充 |
| 2 | `0x48A5` | ChargingCurrent 非零时持续统计充入容量；电流请求归零时使用 `0x0361` 去抖，达到 128 次仍未满足阶段条件则放弃本轮 |
| 3 | `0x48FB` | 清会话计数并转入满电等待 |
| 4 | `0x4907` | 等待满电终点：queued ChargingCurrent 归零后连续计数 64 次，并要求 `XRAM[0x0347] == 100`、滤波电流候选 `0x04AE:0x04AF < 0x0168` |
| 5 | `0x496E` | 从 `0x0388:0x0389` 计算候选 factor，动态限幅后提交到 `0x0342`，置发布 pending 位 |
| 6 | `0x49B6` | 等发布 pending 位被消费者清除；随后设置 `0x0575=8`、`0x91F3=8`，进入同步/重发布窗口 |

这里的“连续 64 次”是调度次数，不应在没有测出调用周期前换算成秒。任一关键门不成立时，
状态机会清计数、退回等待或进入状态 7/idle，而不是立即接受一个瞬时满电读数。

### 开始校正的条件

状态 0 的机器码能确认以下硬条件：

- bank0 veneer `CODE:0x1B40 -> 0xA6C3` 返回成功。真实目标读取
  `XRAM[0x0834] | XRAM[0x0835]`。bank0 charger queue `CODE:0xCB0A` 会把这两个字节作为
  word data 发送命令 `0x14 ChargingCurrent`，所以这里要求排队的充电电流目标非零；
- `XRAM[0x0514] >= 12`；真机当前该字节为 80，且与当前 RSOC 一致，但仍需时间序列证明
  它在所有阶段都严格等于 RSOC；
- `XRAM[0x03A2].bit3 == 1`；
- 从 `0x0363:0x0364` 或 `0x0384:0x0385` 选择的 pack-voltage 镜像高于
  `0x03D2:0x03D3` 高电压阈值。

真机同时采样时，`0x0384:0x0385` 与公开 `EC[0x0438:0x0439]` 都是 16560 mV；更早的
插电基线中 `0x0363:0x0364` 与 pack voltage 都是 16584 mV。阈值
`0x03D0:0x03D1 = 13250 mV`、`0x03D2:0x03D3 = 14820 mV`，按 4S 分别约为
3.3125 V/cell 和 3.705 V/cell。因此这些字段可以从“容量基准”改名为电池包电压工作镜像。

状态顺序也随之明确：校正从一次高电压且仍有 ChargingCurrent 请求的起点建立，状态 1
清空本轮充入容量积分，状态 2 在 ChargingCurrent 持续非零时统计高电压尾充，状态 3/4
等待充电电流归零和满电稳定。`0x03D0=13250 mV` 的低电压分支更像异常恢复/低压入口，
不是正常状态 0 高压启动后必须跨越的放电终点。

因此不需要先完成深度放电才能触发这套 EC 校正。更准确的触发条件是：电池在高于
14820 mV 时进入有效充电，随后完成一段可计量的尾充并自然终止。用户观察到的一次“满电
循环”即使从中高 SOC 开始，也足以让 `0x0388` 累计约 242 mAh 并更新 FCC。

状态 0 还调用 `0x47BA`，从上次公开 FullChargeCapacity 计算两个动态边界。原始指令表现为
一个换算结果的低字节 `+3` 和高字节 `-3` 后分别保存到 `0x039B`、`0x039C`。这说明单轮
校正受到相对旧值的步进限制，但 helper 的定点格式尚未完全恢复，不能简单说成正负 3%。

### 满电终点条件

状态 4 不是只看 Windows 的 100%：

- bank0 veneer `CODE:0x1B52 -> 0xA6CE` 要求
  `XRAM[0x0834] | XRAM[0x0835] == 0`，即 queued ChargingCurrent 已经归零；
- `0x0361` 每次通过检查加一，到 `0x40` 才继续；
- `XRAM[0x0347]` 必须等于 100；当前 RSOC 80 时，现场 `0x0347=80`，强烈支持它是内部
  RSOC 镜像；
- 从公开缓存 `0x04AE:0x04AF` 读取的 16 位量必须小于 `0x0168 = 360`。其他调用者还把
  同一字段与 `0x023A = 570` 比较，并存在滤波后写回路径；结合满充终点用途，它高度疑似
  电流幅值/滤波电流，单位很可能是 mA，但对应的原始 SMBus 命令仍待确认。

所以触发点更准确地描述为：**先有有效统计会话，随后在 100% 且终点量足够低的状态下稳定
保持 64 个调度周期**，才提交校正结果。

## 二进制证据

### `CODE:0x4433` 不提供容量因子

原始字节明确构造命令 `0x1C`、`0x1E`、`0x43`。`0x43` 成功后，以
`MOV DPTR,#0x0343` 调用复制 helper，把返回 word 写入 `XRAM[0x0343:0x0344]`。随后
`CODE:0x69F6` 只把该 word 发布到 `EC[0x04A6:0x04A7]` CycleCount。

该函数没有命令 `0x42`，也没有写 `XRAM[0x0342]`。旧文档把相邻地址 `0x0342` 误认为
命令号，并据此推导出“厂商命令 0x42 提供容量因子”，该推导不成立。

### `CODE:0x4970` 是容量比例校正核心

按原始 DPTR 立即数（不要采用旧 `.d52` 偏小 1 的标签）可还原为：

```c
candidate = capacity_ratio_helper(XRAM16[0x0388]);

if (candidate >= XRAM[0x039B])
    factor = XRAM[0x039B];
else if (candidate < XRAM[0x039C])
    factor = XRAM[0x039C];
else
    factor = candidate;

XRAM[0x0342] = factor;
XRAM16[0x0340] = factor * 100;
```

两个边界来自本轮开始时按旧 FCC 计算的动态范围。候选量 `0x0388:0x0389` 的上游包含容量
差、方向条件和算术 helper；目前可以确认它是本轮校正统计结果，但还不足以将其量纲直接
命名为 mAh、mA·tick 或百分比。

进一步追踪写入点 `CODE:0x5773` 后，量纲可以明显收窄。该路径维护一个 16 位余数
`XRAM[0x03B2:0x03B3]`，每次用常数 `0x4650 = 18000` 做商/余数分解，再把商按方向加到
`XRAM[0x0388:0x0389]` 或 `XRAM[0x038A:0x038B]`。结构等价于：

```c
remainder += per_tick_quantity;
delta = remainder / 18000;
remainder %= 18000;

if (direction_a)
    accumulator_0388 += delta;
else
    accumulator_038a += delta;
```

`18000 = 3600 / 0.2`。如果上游 `per_tick_quantity` 是 mA 且该 worker 每 0.2 秒执行一次，
这正是 `mA * 0.2s / 3600 = mAh / 18000` 的定点库仑积分。方向判断会分别更新
`0x03B5/0x03B6` 两个饱和计数；随后外部 ROM helper `0x7B9E` 分别作用于 `0x0388` 和
`0x038A`。公式结构和常数是高置信度，但 `0x7B9E` 不在当前 flash bank（对应区域为
`0xFF`），所以暂不能断言两个地址就是独立的充/放 mAh 累加器，仍需恢复 ROM helper 或
用动态差分验证。

2026-07-21 的短时拔电记录已经完成这项动态验证。在 237 秒窗口内：

```text
0x0388: 242  -> 242    (+0)
0x038A: 1297 -> 1327   (+30)
RemainingCapacity: 3200 -> 3160 mAh (-40)
pack voltage: 16584 -> 16290 mV
```

窗口前约 20 秒仍插电，之后为电池放电；RemainingCapacity 本身按较粗粒度刷新，所以 `+30`
与 `-40` 不要求逐字节相等。随后自然充电时，`0x0388` 和 `0x038A` 又同时增加；因此更
准确的解释是 `0x0388:0x0389` 只累计充入容量，而 `0x038A:0x038B` 在充电和放电时都会
累计，属于绝对吞吐量/总库仑量。两者单位均为 mAh。状态 5 从 `0x0388` 计算 factor，说明
最终 FCC 主要根据高电压尾充阶段实际充入的容量校正。

### `CODE:0x5A8E` 用因子缩放 LastFullCapacity

初始化/发布流程先处理 `0x0342` 与镜像 `0x03C0` 的变化，随后以常数 100 做乘除运算，
将内部容量按 factor 缩放后写到 `EC[0x0404:0x0405]`。状态 5 置位 `0x03A1.bit1` 请求重新
发布；状态 6 等待 pending 位被消费，再设置两个值为 8 的同步计时器。另一个公开数据同步
入口 `CODE:0x3BD7` 会把 DesignCapacity、FullChargeCapacity、RemainingCapacity 搬到
`0x0A59:0x0A5E` staging 区，通过共享同步调用后再原子式搬回公开 `0x04xx` 字段。

因此报告链为：

```text
状态5提交 factor 0x0342
  -> 置 0x03A1.bit1 发布请求
  -> 0x5A8E 重算 EC[0x0404:0x0405]
  -> 0x3BD7 经 0x0A59:0x0A5E staging 同步公开电池数据
  -> ACPI/Windows 下一次刷新得到新的 FullChargeCapacity
```

这证明 `0x0342` 会影响操作系统看到的 FullChargeCapacity，但不能证明它改变了 Fuel Gauge
内部 learned FCC。

同一流程在缓存不一致时会重新发布 DesignCapacity、DesignVoltage、RemainingCapacity 等
字段；它是遥测初始化/镜像同步，不是用户校准入口。

## 没有找到的逻辑

本次在主 EC 二进制中仍没有确认以下行为：

- 对标准 SBS `ManufacturerAccess (0x00)` 写入学习、复位或校准子命令；
- EC 将本轮 `0x0388:0x0389` 的精确量纲定义为库仑积分；状态机明确累计数据，但积分
  helper 的单位尚未恢复；
- 主机邮箱或公开 EC RAM 中可启动电池校准的命令位；
- EC 把新的容量或 cycle count 写回 Fuel Gauge。

“没有找到”不等于外部 Fuel Gauge 没有自动学习。Fuel Gauge 固件不在本 EC 镜像内，若其
按自身算法在完整充放电过程中更新 learned FCC，主 EC 只能在后续轮询中看到结果。

## 实机验证建议

使用专用只读记录器：

```bash
sudo python3 tools/watch_battery_calibration.py \
  --interval 1 \
  --output battery-calibration-20260721.csv
```

工具每个周期写一行 CSV 并立即 flush，Ctrl-C 或 SIGTERM 会正常关闭文件。它不包含任何
I2EC write 路径。建议：

- 完整循环用 1 秒间隔，约每天 86400 行，文件大小可控；
- 接近 95% 后改用 0.1 至 0.2 秒间隔，捕获 64 次连续确认和状态 4 到 6 的转换；
- 不要同时运行其他 `/dev/port` 或 I2EC 工具，记录器会独占锁定设备；
- 另记插拔 AC、开始放电、到达最低电量、开始充电、Windows 显示 100% 和 FCC 改变的
  本地时间；这些事件用于和 CSV 的 `timestamp` 对齐；
- 一次实验从放电前开始，持续到 FCC 改变后至少 10 分钟。不要为了实验写 XRAM 或强制
  改 RSOC，这会破坏状态机的自然条件。

记录器连续采集以下字段：

| 地址 | 用途 |
|---|---|
| `0x0340:0x0342` | 比例派生值和最终 factor |
| `0x0361:0x0362` | 阶段去抖/连续条件计数 |
| `0x0384:0x0385` | pack-voltage 工作镜像 |
| `0x0388:0x0389` | 校正候选统计 |
| `0x039B:0x039C` | 动态限幅边界 |
| `0x03A1:0x03A3` | 发布 pending、条件位和状态号 |
| `0x03C0` | factor 镜像 |
| `0x0402:0x0405` | DesignCapacity / FullChargeCapacity |
| `0x0436:0x0439` | RemainingCapacity / Voltage |
| `0x04A6:0x04A7` | CycleCount |

### 如何从 CSV 判定阶段

1. 找 `state` 首次从 0 变为 1：这是本轮校正会话真正开始，而不是简单的“开始放电”。
2. 在状态 1/2 中画出 `charged_capacity_0388`、`throughput_capacity_038a`、`pack_voltage_0384`、
   `remaining_capacity` 和时间的关系；用一阶差分判断候选量按时间、电量变化还是电流变化。
3. 找状态 4 中 `debounce_0361` 连续增长到 64 的区间，同时核对 `rsoc_mirror=100` 和
   `end_point_value<360`。
4. 找状态 5 的 `factor`、`limit_high_039b`、`limit_low_039c`，确认候选值是否被限幅。
5. 比较 `factor`、`factor_mirror`、`full_charge_capacity` 的变化顺序，再观察状态 6、
