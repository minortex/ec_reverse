# 2026-07-19 22:37 CST：17000 mV 充电目标现场验证

## 结论

本机当前 `17600 mV -> 17000 mV` 的 EC 充电目标降额，直接原因是
`XRAM[0x09C9:0x09CA] = 0x2BDF` 落入 high-voltage stress 的
`150 mV/cell` 档，而不是循环次数、Normal/Health 模式或当前已确认的辅助条件。

`XRAM[0x09C7]` 和 `XRAM[0x09C8]` 是形成该 stress word 的两级计数器；它们不直接参与
最终电压减法。最终选择档位的是 `0x09C9:0x09CA`。

当前为 4S，固件计算与实测完全一致：

```text
base target       = 17600 mV
stress            = 0x2BDF = 11231
stress tier       = 150 mV/cell
pack derating     = 4 * 150 = 600 mV
EC desired target = 17600 - 600 = 17000 mV
```

这里确认的是 EC desired target 及 charger 队列目标。当前没有外部 charger 的电压回读，
因此不能仅凭这些寄存器证明模拟充电环路的实际输出精确等于 `17000 mV`。

## 现场寄存器证据

2026-07-19 使用 dedicated I2EC 只读路径采集：

| 地址 | 读值 | 解释 |
|---:|---:|---|
| `0x200D` | `0xCB` | I2EC 当前为 read-write；本次只执行读取 |
| `0x030E:0x030F` | `44 C0` | 大端基准 `0x44C0 = 17600 mV` |
| `0x0438:0x0439` | `6C 41` | 小端包电压 `0x416C = 16748 mV` |
| `0x0490` | `0x07` | AC 基线时 bit 0/1/2 全部为 1 |
| `0x0491` | `0xC0` | bits 7:6 为 `11b`，即 4S |
| `0x04A2:0x04A3` | `E0 0B` | 小端温度原始值 `0x0BE0`，约 30.85 C |
| `0x04A6:0x04A7` | `39 00` | 循环次数 57 |
| `0x0522:0x0523` | `68 42` | 小端 desired target `0x4268 = 17000 mV` |
| `0x0497` | `0x45` | bit 0 为 1，`D1BB` 电池数据会话 gate 开放 |
| `0x05B9` | `0x00` | 独立状态路径观测值；不是 `D1D1` 直接 gate |
| `0x07A6` | `0x00` | Normal 模式，无模式最低降额 |
| `0x0832` | `0x00` | 稳态无待处理 ChargingVoltage 请求 |
| `0x0836:0x0837` | `68 42` | queued target 同为 `17000 mV` |
| `0x09C9:0x09CA` | `2B DF` | 大端 stress `0x2BDF = 11231` |
| `0x0A73:0x0A75` | `00 02 01` | 稳态 SMBus 事务结果为成功 |

档位比较是严格大于门槛：

```text
0x21C0 < 0x2BDF <= 0x2D00
```

因此命中 `150 mV/cell`，尚未命中 `200 mV/cell`。

## 排除其他已知选档来源

`CODE:0xD1D1` 从最高档向下判断多个 OR 条件。当前现场状态为：

- 循环次数为 57，远低于 150 档的 `>= 350` 门槛；
- `0x07A6 = 0x00`，Normal 模式不提供最低降额；
- 本机构建的 aging 调用路径局部辅助 seed 不足以达到 150 档门槛 13；
- 未降额基准稳定为 `17600 mV`，不是基准本身下降到 `17000 mV`；
- `0x0522:0x0523` 与 `0x0836:0x0837` 同为 `17000 mV`，排除了旧值尚未进入
  charger 队列的解释。

所以在当前已恢复的 `D1D1` 决策逻辑中，只有 `0x2BDF` 能解释 150 档，而且计算结果与
目标精确相等。不能把这句话扩大为“外部 charger 实际电压只有这个可能原因”；PD 输入、
charger 硬件保护和模拟环路仍可能影响实际电压，但它们不是本次 EC desired target 从
`17600` 变成 `17000` 的原因。

## Stress 的动态形成证据

### 三层计数关系

这里有三个不同的量，不能把它们都理解成“stress 时间”：

```text
每次 D1D1 维护调用
        |
        +-- 09C7 sample_counter：短周期计数，达到 60 后回零
                    |
                    +-- 09C8 window_counter：合格高压窗口计数，达到 60 后回零
                                |
                                +-- 09C9:09CA stress：只在这里加 +1、+3 或 +7
```

具体过程是：

1. `D1D1` 被周期维护任务调用一次。只有电池包电压达到
   `series_cells * 4100 mV` 时，当前调用才算合格高压样本；4S 门槛是 `16400 mV`。
2. 合格样本使 `0x09C7` 加一。达到 60 后，`0x09C7` 清零，`0x09C8` 加一。
3. 达到第二个 60 后，`0x09C8` 清零，读取温度并给 `0x09C9:0x09CA` 增加一次权重：
   `+1`、`+3` 或 `+7`。
4. 如果包电压低于 4.10 V/cell，该窗口不会增加 stress；`0x0490.bit1` 为 0 则更强，
   会直接把四个字节全部清零并返回。

温度只决定“每个合格窗口加多少”，不直接把目标电压改成某个值：

| `04A2:04A3` 原始温度 | 约摄氏温度 | 每个合格窗口对 stress 的增量 |
|---:|---:|---:|
| `< 0x0BD6` | `< 29.85 C` | `+1` |
| `0x0BD6..0x0C3A` | `29.85..39.85 C` | `+3` |
| `> 0x0C3A` | `> 39.85 C` | `+7` |

这里的 `+1/+3/+7` 是内部 stress 分数，没有 mV 单位。每次加分通常不会让 target 发生
变化；只有累计值严格跨过档位门槛时，下一次 `D1D1` 计算才会切换降额档：

```text
stress <= 0x10E0  ->   0 mV/cell（假设没有其他 OR 条件）
stress >  0x10E0  ->  50 mV/cell
stress >  0x1950  -> 100 mV/cell
stress >  0x21C0  -> 150 mV/cell
stress >  0x2D00  -> 200 mV/cell
stress >  0x3DE0  -> 250 mV/cell
```

当前 `0x2BDF` 已超过 150 档门槛，但尚未严格超过 `0x2D00`，所以目标保持 17000 mV。
到下一档最少还需要：

```text
0x2D01 - 0x2BDF = 290 stress points
```

如果以后始终保持当前 `+3` 档，需要 97 次完整的两级窗口才会首次超过 `0x2D00`。按本次
真机观察到的约 60.5 分钟/次计算，约等于 97.8 小时合格高压时间。跨过后，在其他条件
不变时，4S 目标会从 `17600 - 4*150 = 17000 mV` 变为
`17600 - 4*200 = 16800 mV`。

例如一次现场输出：

```text
... temp=30.85C ... stress=0x2BDF tier=150 counters=59/40 ...
```

其中 `counters=59/40` 的含义是：

```text
09C7 sample_counter  = 59
09C8 window_counter  = 40
09C9:09CA stress     = 0x2BDF
```

下一次达到 60 后可能看到：

```text
counters=0/41
```

这表示短计数器回零、第二级计数器加一；这次不会改变 stress。只有第二级也从 59 进位到
0 时，才会看到类似：

```text
counters=0/0 stress=0x2BDF -> 0x2BE2
```

在约 30--40 C 温度档，这个变化量是 `+3`。反汇编中的 helper 先把权重加到低字节
`0x09CA`，再把进位加到高字节 `0x09C9`，所以 `0x2BDF + 3 = 0x2BE2`。

### 监控命令输出逐字段解释

例如：

```text
2026-07-20T20:00:00.000+08:00  AC=1 0490=0x07 gates=1/1/1 cells=4
    pack=16748mV base=17600mV target=17000mV queued=17000mV
    temp= 30.85C cycles= 57 mode=0x00 stress=0x2BDF tier=150 counters=59/40
    pending=0x00 SMBus=00/02/01 aux=0x02
```

各字段含义如下：

| 输出字段 | 对应地址/含义 |
|---|---|
| `AC=1` | `0490.bit0=1`，当前 AC 状态；`0` 是电池供电 |
| `0490=0x07` | 电源/aging gate 原始字节 |
| `gates=1/1/1` | 依次是 `0490.bit0/bit1/bit2` |
| `cells=4` | 从 `0491.bits7:6` 解出的串联电芯数 |
| `pack` | `0438:0439` 的实时包电压 |
| `base` | `030E:030F` 的未降额基准 |
| `target` | `0522:0523` 的 EC desired target |
| `queued` | `0836:0837` 的 charger 队列镜像，不是实际电压回读 |
| `temp` | `04A2:04A3` 转换后的摄氏温度 |
| `cycles` | `04A6:04A7` 的循环次数 |
| `mode` | `07A6`，bits 5:4 是 Normal/Balanced/Health 模式贡献 |
| `aux` | `0A54` 共享 scratch，可能在 worker 中短暂变化，不能单次定性 |
| `stress` | `09C9:09CA`，最终参与档位选择的累计值 |
| `tier` | 仅按 stress 门槛计算的档位，不包含循环/模式等其他 OR 条件 |
| `counters` | `09C7/09C8`，即 sample/window 两级计数器 |
| `pending` | `0832`，bit1 表示 ChargingVoltage 请求待处理 |
| `SMBus` | `0A73/0A74/0A75`，worker 的结果/状态/成功标志；事务中可能是瞬态值 |

默认监控器忽略 `aux` 和 SMBus 中间状态，只在关键字段变化时输出。需要查看这些共享
工作区瞬态时，加上 `--transients`：

```sh
sudo python3 tools/watch_i2ec_battery.py --interval 0.1 --transients
```

连续两次读取曾捕获：

```text
第一次：09C7/09C8 = 3B/18，09C9/09CA = 2B/DC
第二次：09C7/09C8 = 16/19，09C9/09CA = 2B/DF
```

这对应：

1. `0x09C7` 达到 60 后回绕；
2. `0x09C8` 从 `0x18` 增加到 `0x19`；
3. 约 30--40 C 温度档使 stress 从 `0x2BDC` 增加 `+3` 到 `0x2BDF`。

原始机器码中的 16 位 helper 先给低字节 `0x09CA` 加 `1/3/7`，再把进位加到高字节
`0x09C9`。本次实测与该顺序逐项一致，证明 stress accumulator 当前正在运行。

监控中还观察到 `0x09C7` 的有效递增间隔约为 `1.008 s`。这与此前根据相邻 IT5570
时钟资料推算的约 720 ms 不一致；真机有效周期应优先作为当前平台结论。按本次节拍，
两级 60 次计数约每 60.5 分钟给 stress 增加一次温度权重。节拍差异仍需继续审计 IT5571
时钟和调度分频来源。

## AC 拔插实验

使用 `tools/watch_i2ec_battery.py --interval 0.1` 完成一次只读 AC 拔插：

```text
初始 AC：0490 = 0x07
拔出 AC：0490 = 0x06
插回 AC：0490 = 0x0F
```

拔出 AC 只清除了 `0x0490.bit0`；bit 1 和 bit 2 始终为 1。整个过程中：

```text
stress        = 0x2BDF
desired target = 17000 mV
queued target  = 17000 mV
```

`0x09C7/0x09C8` 也从 AC 状态连续运行到电池状态，再连续运行到 AC 恢复，没有清零。
拔出 AC 后包电压仍约 `16642--16665 mV`，高于 4S high-voltage 门槛 `16400 mV`，因此
计数器继续推进符合固件逻辑。

由此确认：普通 AC 拔插不能触发 stress 清零。`CODE:0xD1D1` 只有实际采到
`0x0490.bit1 == 0` 时，才会把 `0x09C7--0x09CA` 清零并返回。需要电池会话状态机产生并
维持足够长的 bit 1 清零窗口，而不是只改变 AC bit 0。

## 为什么当前值未必代表完整历史

固件复位审计没有发现启动清零覆盖 `0x09C7--0x09CA`：

- C51 初始化表在当前镜像中为空；
- reset 清零范围不包含这四个字节；
- 没有发现从 Flash、EEPROM 或电池缓存恢复该 word 的写入路径；
- 运行期写入只来自 bit 1 清零分支、两级计数器和 `+1/+3/+7` helper。

因此 `0x2BDF` 的算法用途确定是 high-voltage stress，但其起点可能包含未初始化 SRAM
内容。当前实测证明它正在继续累计，不能反向证明全部 `11231` 都来自真实历史高压时间。

## 监控工具

默认只输出电源、目标、stress、计数器和队列等关键变化：

```sh
sudo python3 tools/watch_i2ec_battery.py --interval 0.1
```

需要同时观察共享 scratch 和 SMBus worker 中间状态时使用：

```sh
sudo python3 tools/watch_i2ec_battery.py --interval 0.1 --transients
```

监控器只调用 I2EC read，不写 EC memory。当前 `SPCTRL1=0xCB` 仍表示全局 I2EC
read-write，因此运行监控时不得同时执行 `i2ec_rw.py write` 或其他直接访问
`0x681--0x683` 的工具。
